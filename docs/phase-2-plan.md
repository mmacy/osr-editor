# Phase 2 plan — the map editor

Implementation plan for phase 2 of [the osr-editor spec](spec.md). Phase 2 delivers the graph-paper canvas and the full geometry tool set — rooms, corridors, edges and doors, areas, entrance, transitions, multi-level and multi-dungeon management — plus live structural lint on the map and the `GeometryImporter` protocol with import-from-project as its first implementation. Milestone: author a two-level dungeon's complete geometry from a blank grid, lint clean, and play its empty halls in osr-web.

Five research facts shape this plan more than any others. First, **osrlib stores edges only under the canonical north/west key and silently ignores everything else** — `edge_key` folds a cell's south and east edges onto the neighbouring cell's north/west (`osrlib/crawl/dungeon.py:118-137`), `LevelSpec.edge` consults only that form (`dungeon.py:526`), and `validate_adventure` never inspects `edges` at all — so a malformed, non-canonical, or out-of-bounds key is representable in a fully valid document and is dead data forge calls "the single most dangerous silent failure in the loop" (`osrforge/check.py:151`). The editor answers twice: the `SetEdges` op refuses to author anything but canonical in-bounds keys, and the `edge_invalid` lint catches what foreign documents bring in. Second, **forge's `check.py` is a complete, quotable specification for the five shared lint checks**: reachability is a BFS over one of two graph flavors — *inclusive* (open edges plus doors in any state, secret included) and *non-secret* (secret doors become walls) — seeded from each dungeon's **first entrance-bearing level only** ("an override-authored second entrance must not manufacture phantom reachability", `check.py:117-119`), expanding over passable edges and directed transitions; only `stairs_up`/`stairs_down` must pair ("trapdoors and chutes are one-way by osrlib's design", `check.py:247-248`); and severity is a producer-pinned table (`check.py:62-70`). The editor mirrors these semantics and messages exactly, which is what makes the spec's "forge's finding vocabulary where the checks coincide" true rather than aspirational. Third, **the content models leave two geometry mistakes legal**: area-cell overlap is unchecked anywhere (`area_at` silently resolves to the first area in authored order, `dungeon.py:528-540`), and duplicate area ids are only a validation finding — which is exactly why the spec adds the editor-only `area_overlap` lint and why `CreateArea` refuses a duplicate id at commit. Fourth, **`TransitionSpec` carries no validators beyond `to_level_number >= 1`** (`dungeon.py:213-227`) — one-way-ness is authored convention, the runtime treats all four kinds identically (`exploration.py:967-975`), and dangling targets are ordinary diagnostics — so transition ops validate the *source* cell and leave targets to validation, which is what lets an import land stairs to a level that doesn't exist yet. Fifth, **phase 2 adds zero dependencies**: the canvas is bespoke by design, importer discovery is stdlib `importlib.metadata` entry points, and the lint engine is pure Python over models already in hand.

## Scope

In scope:

- `ops.py` growth — the geometry ops (`SetEdges`, `SetEntrance`, `CreateArea`, `SetAreaCells`, `SetAreaField`, `RemoveArea`, `AddTransition`, `RemoveTransition`) and the management ops (`AddDungeon`, `SetDungeonField`, `RenameDungeon`, `RemoveDungeon`, `AddLevel`, `RenumberLevel`, `ResizeLevel`, `RemoveLevel`), joined into `AnyEditOp`
- `documents.py` growth — op application for the new vocabulary, precise delta pointers, the rename/renumber cascades
- `lint.py` — the tier-3 structural lint: the five forge-vocabulary checks mirrored exactly, plus editor-only `area_overlap`; `Finding` grows `severity`; the address grammar grows `cell:` and `edge:` segments
- `importers.py` — the `GeometryImporter` protocol, the `ImportedGeometry` payload, entry-point discovery under `osreditor.importers`, and the built-in import-from-project importer registered through that same seam
- `app.py` growth — importer routes (`GET /api/importers`, sniff, load), the `op_invariant` error mapping with structured offenders, importer error mappings
- Frontend — the map editor: canvas renderer in the graph-paper design language, edge-key mirror, hit testing, gesture reducers, the seven tools, the door and area inspectors, level tabs and dungeon switcher, management dialogs, lint rendered on the map, diagnostics click-to-navigate down to cells and edges, the import dialog
- The committed doors-and-transitions torture fixture (the spec's round-trip fixture pair, geometry half)
- Type generation updates and the test suites for all of the above, green in CI

Out of scope (each with the phase that picks it up): area content — encounters, features, traps, treasure, the keyed-entry cards, stocking badges and hollow/filled key numbers, and the wandering inline-table form (phase 3; phase 2's area inspector covers identity only — id, name, description); the monster editor (phase 4); forge-backed projects entirely, including op→override translation, the `geometry:` override emission for imports into workdirs, and the cross-implementation lint-parity test that runs forge's `check()` beside the editor's lint on shared fixtures (phase 5, when `osr-forge` becomes a dependency; phase 2 pins parity by mirroring forge's documented semantics, messages, and severities, cited to `check.py` line by line); conversion (phase 6); aids (phase 7); walk mode (phase 8); docs site and release engineering (phase 9). Sidecar view state (zoom, pan, active level) stays unwritten — the map keeps view state in memory, because an always-saved editor must not turn every pan into a disk write, and no feature consumes persisted view state yet; the sidecar envelope is untouched this phase, and persisted view state is deferred to phase 5, whose review queue (resuming correction work across sessions, per-flag navigation) is the first surface where restoring where-you-were earns the writes — the sidecar's additive-only envelope lets it land then without ceremony. The small-complete-module round-trip fixture waits for phase 3's content tools; phase 2 commits the geometry torture case. Watabou One Page Dungeon import is future work riding the seam shipped here.

## Work items

### 1. Geometry ops

Frozen `EditOp` subclasses in `ops.py`, each targeting a level by `dungeon_id` + `level_number` (`ge=1`) exactly as `SetWandering` does, with `op_target_not_found` on a miss. Editor-enforced rejections beyond model validation raise the new `OpInvariantError` (work item 4). The op-level philosophy, continuing phase 1's `travel_turns >= 0` guard: **reject what is never intentional and is not a transient editing state** (malformed edge keys, out-of-bounds cells, duplicate ids); **admit what a workflow legitimately passes through** (dangling transition targets, missing entrances) and let diagnostics guide the fix.

- **`SetEdges`** — `edges: dict[str, Edge | None]`, min one entry: a batch of edge-key → assignment, `None` deletes the entry (wall). A *non-`None`* assignment validates its key against the canonical grammar `^(0|[1-9][0-9]*),(0|[1-9][0-9]*):(north|west)$` — canonical side only, non-negative, no leading zeros (forge's aliasing guard, `contracts/overrides.py:54-56`, tightened to north/west because the editor authors storage form, not override input) — and requires both incident cells in bounds (the key's cell and its north or west neighbour; osrlib renders any other entry as wall without consulting it, `dungeon.py:524-526`). A *`None`* assignment instead accepts any key string that exactly matches an existing entry in the level's `edges` map (and rejects a key matching nothing): deletion authors no key, so the editor-never-writes-a-bad-key invariant holds, while the malformed, non-canonical, and out-of-bounds keys a foreign document legally carries stay deletable — the `edge_invalid` remediation path (work item 5) and the replace-mode import's clear-all step (work item 10) both depend on exactly this. An explicit `Edge(kind=WALL)` value is rejected: deletion is the one way to say wall, so editor-written documents keep a single representation (foreign documents legally carry explicit wall entries — forge's override merge writes them — and they render fine; the editor just never authors the redundant form, and touching one with the wall tool deletes it). The `Edge` model itself enforces the door coupling (`door` present iff `kind="door"`, `dungeon.py:206-210`), so door edits are just `SetEdges` with a new `DoorSpec`.
- **`SetEntrance`** — `entrance: Position | None`. A position must be in bounds; `None` clears, and the resulting `entrance_missing` finding is the legal, navigable consequence.
- **`CreateArea`** — `area_id`, `cells` (min 1, all in bounds), optional `name`/`description` (default `""`; import and the room tool both use them). Rejects an empty `area_id` and an id already present on that level — a duplicate is never intentional (the model can't see it and `validate_adventure` would only flag it later; same reasoning the spec applies to monster-id collisions). Cells may overlap another area's — that is `area_overlap` lint's territory, legal by the model and sometimes transiently useful mid-edit.
- **`SetAreaCells`** — replace an area's cell cluster wholesale (min 1, all in bounds). The paint/lasso tool's op.
- **`SetAreaField`** — `field: Literal["id", "name", "description"]` with a pairing validator like phase 1's field ops. `id` is the re-key affordance (key numbers render on the map, so re-keying is a map concern): rejects empty and duplicate ids like `CreateArea`. Nothing references area ids in the document, so re-keying cascades nowhere. Phase 3 does not grow this literal — encounter, trap, and treasure have their own ops per the spec's vocabulary.
- **`RemoveArea`** — removes the `AreaSpec`; its cells become corridor (osrlib's convention — cells in no area). Removing an area discards any content it carries; the frontend confirms when content exists (foreign documents), silently otherwise.
- **`AddTransition`** — carries a full `TransitionSpec`. The source `position` must be in bounds and unoccupied — osrlib's `transition_at` returns the first match (`dungeon.py:542-554`), so a second transition on one cell is dead data; editing a transition is a remove+add batch (one gesture, one undo step). The target is deliberately unvalidated beyond the model's own `ge=1`: an unknown dungeon, missing level, or out-of-bounds target cell is a validation finding (`transition_target_unknown`, `transition_target_cell_out_of_bounds`), legal while editing — an import may land stairs before their destination level exists.
- **`RemoveTransition`** — by source `position`; `op_target_not_found` if no transition sits there. Removes the *first* matching entry, osrlib's own resolution order (`transition_at` is first-match) — a foreign document can stack two transitions on one cell, and first-match keeps "remove one per existing entry" sequences (the replace-mode import) correct by construction.

The `op_target_not_found` rejection covers every targeting miss in the vocabulary — unknown dungeon, unknown level, unknown area id, no transition at the position — and `OpTargetNotFoundError`'s docstring grows from "a dungeon or level" to match.

Delta pointers are precise and level-scoped: `SetEdges` → `/dungeons/<i>/levels/<j>/edges`, area ops → `.../areas`, `SetEntrance` → `.../entrance`, transition ops → `.../transitions` — the small payloads the spec promised phase 2's high-frequency gestures.

### 2. Dungeon and level management ops

The invariants the spec assigns to the ops — unique ids, unique level numbers, min one of each — enforced at commit with `OpInvariantError`, not discovered at re-validation:

- **`AddDungeon`** — `dungeon_id`, `name` (default `""`), `width`/`height` (`ge=1`): scaffolds level 1 with an entrance at `(0, 0)`, exactly like `starter_adventure`, so a new dungeon validates clean from birth and the entrance tool moves what exists rather than the panel nagging about what doesn't. Rejects a duplicate dungeon id.
- **`SetDungeonField`** — `field: Literal["name"]`; the one plain settable field. The literal grows if `DungeonSpec` ever does.
- **`RenameDungeon`** — `old_id`, `new_id` (non-empty, not already taken). Renaming cascades: the dungeon's `id`, the town's `travel_turns` key (order preserved), and every `TransitionSpec.to_dungeon_id` naming it, across all dungeons — a rename means "same thing, new name", so references follow, atomically, in one undo step. Contrast **`RemoveDungeon`** — rejects removing the last dungeon (`min_length=1` on `Adventure.dungeons`), and deliberately does *not* cascade: a removed dungeon's dangling `travel_turns` entry and inbound transitions are honest diagnostics (`travel_unknown_dungeon`, `transition_target_unknown`) — removal never silently edits other subtrees.
- **`AddLevel`** — `number` (unique in the dungeon), `width`/`height`; no entrance (validation requires one per *dungeon*, not per level). The new level is *inserted* before the first existing level whose `number` exceeds it, appended when none does — deterministic over any tuple, ascending-preserving over an ascending one; **no op ever reorders existing levels**. Stored order is rules-visible when more than one level bears an entrance — `EnterDungeon` lands on the first entrance-bearing level in stored order (`exploration.py:983`), and forge's reachability seeds the same expression — so a re-sort could silently change where a foreign dungeon's play begins. Insertion is safe precisely because a new level carries no entrance; foreign order is never touched, editor-authored dungeons stay ascending unless a `RenumberLevel` chose otherwise, and the tabs and sidebar sort by `number` for display regardless of storage.
- **`RenumberLevel`** — `old_number` → `new_number` (unique); cascades every transition in the document targeting `(this dungeon, old_number)`. Same rename-vs-remove logic as dungeons; per the no-reorder rule it leaves the tuple where it sits (display order is sorted anyway, and moving it could change the stored-order entry semantics above).
- **`RemoveLevel`** — rejects removing a dungeon's last level. Inbound transitions dangle as diagnostics.
- **`ResizeLevel`** — `width`/`height` (`ge=1`). Shrinking below existing content is rejected listing every offender — areas with cells outside the new bounds, features with out-of-bounds cells, transitions whose *source* leaves the grid, an entrance outside — as `details.offenders`, each an address-grammar string plus a message, so the dialog renders a click-to-navigate list. (Transitions *elsewhere* that target now-out-of-bounds cells in this level are not offenders; they become `transition_target_cell_out_of_bounds` validation findings, the spec's dangling-reference rule.) One pinned addition the spec's offender list deliberately omits: **edge entries whose incident cells fall outside the new bounds are pruned by the op**. Edges are spatial annotation, not content — osrlib treats an out-of-bounds entry as nonexistent (`dungeon.py:524-525`) and `validate_adventure` never sees it, so keeping it would strand invisible `edge_invalid` errors on keys the map cannot even display, and rejecting on it would demand the user hand-erase edges they cannot see. The prune is deterministic, documented on the op, and inside the single undo step.

Management deltas are honest about their blast radius: `AddLevel`/`RemoveLevel`/`RenumberLevel`/`ResizeLevel` → `/dungeons/<i>` (indices shift or siblings re-sort), `AddDungeon`/`RemoveDungeon` → `/dungeons`, `RenameDungeon` → `""` (the cascade may touch the town and any dungeon's transitions).

### 3. Op application in the document service

`_apply_op` in `documents.py` grows from a three-way `isinstance` chain into dispatch over the full vocabulary, with two shared helpers replacing the inline dungeon/level walk: resolve a `(dungeon_id, level_number)` target to indices or raise `OpTargetNotFoundError`, and rebuild a level inside its dungeon inside the adventure (the bottom-up frozen-model `model_copy` idiom, unchanged). Everything else phase 1 built stays load-bearing and untouched: the whole-batch re-validation round-trip remains the single model-validity enforcement point (`model_copy` still bypasses validators), atomicity, revision issuance, undo/redo, persistence, and delta coalescing all apply to the new ops for free. The re-validation also backstops every invariant the ops enforce ahead of it — a bug in an op-level check surfaces as `op_rejected`, never a committed invalid document.

### 4. Errors and routes

- **`OpInvariantError`** joins `errors.py`: an editor-enforced semantic rejection — duplicate id, occupied transition cell, last dungeon or level, non-canonical edge key, out-of-bounds cell, explicit wall entry, resize offenders. Maps to 422 `op_invariant`; the message names the violation, and `details.offenders` (present only when a list exists, i.e. `ResizeLevel`) carries `{"address", "message"}` entries in the address grammar.
- **Importer errors**: `ImporterNotFoundError` → 404 `importer_not_found`; `ImportSourceInvalidError` → 422 `import_source_invalid`, wrapping whatever the importer's load raised (an unreadable path, a sniff-negative source, a document that fails `load_adventure`) with the importer's message.
- **Importer routes**, behind the auth dependency like everything (the introspection test extends automatically):

    ```text
    GET  /api/importers                      # registered importers: format_id, label
    POST /api/importers/sniff                # {path} → format_ids that recognize it
    POST /api/importers/{format_id}/load     # {path} → ImportedGeometry
    ```

    Import needs no apply route: the loaded geometry crosses to the frontend, and the import dialog turns the user's choices into one ordinary op batch through `POST /projects/{id}/ops` — undoable, revision-guarded, immediately linted, exactly the spec's "imported geometry lands as ordinary op batches". Paths are absolute (the `_AbsolutePathRequest` validator), read-only, and local — the same honestly-local posture as export, noted in the docstrings. `sniff` never errors: a nonexistent or unrecognized path answers an empty match list (presence-level probing has nothing to throw), and the dialog renders zero matches as an inline "no installed importer recognizes this path" message.

### 5. The structural lint tier

New module `lint.py`, the second producer within the locked `Finding` shape:

- **`Finding` grows `severity: Literal["error", "warning"]`** — additive envelope growth, the rule phase 0 pinned. Forge findings carry severity as a field with a producer-pinned table, and the diagnostics panel needs the distinction (the design language reserves red for errors; warnings use the pencil palette). The validation producer sets `"error"` on every finding — `validate_adventure` output gates publish, which is what error means here.
- **The engine**: `lint_adventure(adventure) -> tuple[Finding, ...]`, pure, whole-document — reachability is inherently cross-level (transitions jump levels and dungeons), so the unit of computation is the document. "Incremental" cashes out as *recomputed live, in-process, on every commit*: a full pass is one BFS plus per-level scans over documents whose largest known instance is ~300 KB — microseconds, measured in the test suite rather than assumed. If a pathological document ever disagrees, the named fallback is memoizing per-level scans keyed on the delta's touched subtrees; nobody builds that speculatively. `compute_diagnostics` populates the `lint` tuple alongside `validation`; the tiers are independent, so lint runs even when validation fails.
- **The five shared checks mirror forge exactly** — semantics, messages verbatim, severities from forge's table (`check.py:62-70`), cited here so the implementer needn't re-derive:
    - `edge_invalid` (error) — per `check.py:150-185`: a key failing the shape regex (`x,y:side`, any compass side) → "edge key `{key!r}` is malformed — expected 'x,y:side'"; a well-formed key that is not its own canonical form → "edge key `{key!r}` is never consulted — osrlib's canonical form is `{canonical!r}`"; a canonical key with an out-of-bounds incident cell → "edge key `{key!r}` references the out-of-bounds cell `{cell}`". Address: the level (an invalid edge is not a navigable edge; the key rides the message). Editor-authored documents can only acquire these from foreign files — `SetEdges` refuses to *write* them — which is why the lint must exist, and the remediation is the diagnostics panel itself: an `edge_invalid` finding row carries a "remove entry" action committing `SetEdges` with the offending key → `None` (the delete rule admits any existing key). The action resolves its key by enumeration, never extraction — the phase 1 amendment's rule: the builder recomputes the level's invalid keys client-side (the edge-key mirror has the parse), renders each key's message exactly as the lint does, and acts on the key whose rendering equals the finding's message; no match (a stale finding) returns no ops. So the one error-severity finding only foreign documents produce is a click to fix, and a hostile key can at worst decline the click.
    - `area_unreachable` (error) — no cell of the area is in the **inclusive** reachability set (BFS from each dungeon's first entrance-bearing level's in-bounds entrance, over open edges and doors in any state including secret, plus directed in-bounds transitions into levels that exist; `check.py:110-147, 194-205`). Message "no path from any entrance reaches this area"; address the area.
    - `orphan_cell` (warning) — a cell in no area, not inclusively reachable, with at least one non-wall edge ("renders as corridor but no path reaches it"; blank bounding-box cells stay silent, `check.py:207-226`). Address: level + `cell:` segment — richer than forge's level-only location, same finding.
    - `secret_only_access` (warning) — reachable inclusively but not in the **non-secret** flavor (secret doors as walls): "every path into this area passes through a secret door" (`check.py:228-241`). Address the area.
    - `transition_unpaired` (warning) — `stairs_up`/`stairs_down` only; paired iff the target level exists and has a transition at `to_position` pointing back at this `(dungeon, level, position)` (`check.py:243-268`). Message mirrors forge's; address: level + `cell:` segment for the source cell.
- **`area_overlap`** (warning, editor-only): for each unordered pair of areas on a level sharing at least one cell, one finding addressed to the pair's later area in authored order (the one `area_at` silently loses, `dungeon.py:528-540`), message naming both areas, the shared-cell count, and one sample cell. Runs after the mirrored vocabulary, per level.
- **Ordering is deterministic** and mirrors forge's exactly: findings are **grouped by check id in the vocabulary's order** (`check.py:189`), each check doing its own full dungeon/level sweep in document order — `edge_invalid`, `area_unreachable`, `orphan_cell` (bounding-box scan, y outer), `secret_only_access`, `transition_unpaired` — with `area_overlap`, the editor extension, as a sixth group after them. Not per-level interleaving. Stable output keeps tests exact and the panel steady.
- **The address grammar grows two segments**, additive within the phase 1 grammar: `cell:<x>,<y>` and `edge:<x>,<y>:<north|west>`, appended after the level segment. Values are numeric grammar, unambiguous without encoding; the general segment parse rule stays "kind is up to the first `:`, value is the remainder" (already true — percent-encoding makes ids colon-free). The `edge:` segment ships with the grammar (the inspector and hover refs use edge addressing) even though no phase 2 finding emits it — `edge_invalid` deliberately addresses the level.
- **Parity containment**: the mirrored semantics are pinned by fixture tests in this repo; the cross-implementation test that runs forge's `check()` beside `lint_adventure` on shared fixtures arrives with phase 5, when `osr-forge` joins the dependencies and forge-backed review makes the comparison natural. Until then the citations above are the contract, and the torture fixture (work item 10) is written to be reusable as that shared case.

### 6. The geometry importer seam

New module `importers.py`:

- **The protocol**, runtime-checkable like `ProjectStore`: `format_id: str`, `label: str`, `sniff(path: Path) -> bool` (cheap, presence-level, never loads), `load(path: Path) -> ImportedGeometry` (raises `ImportSourceInvalidError` with a human message on anything unloadable).
- **`ImportedGeometry`**, the editor-defined frozen payload (wire-visible, so typegen covers it): optional `title` and `description` offered for adoption; `levels` (min 1) of `ImportedLevel` — `label` (source-side display name), `width`, `height`, `edges: dict[str, Edge]` (canonical keys — the importer owns normalization), `areas` (id, name, description, cells), `entrance: Position | None`, `transitions: tuple[TransitionSpec, ...]`, and `notes: tuple[str, ...]` (the importer flagging what it guessed, rendered in the dialog). Phase 2's dialog ignores `title`/`description` — adoption UX belongs with a converter whose sources carry meaningful metadata (Watabou); the payload is complete now so that converter is an installable package, not an editor change.
- **Discovery**: `importlib.metadata.entry_points(group="osreditor.importers")`, each entry a zero-arg callable returning an importer instance. A broken entry point logs a warning and is skipped — a third-party package must never break boot. The registry is built once per app in `create_app()`.
- **Import-from-project**, the built-in first implementation, registered through the same entry-point group in the editor's own `pyproject.toml` — dogfooding the seam is what keeps the protocol honest, per the spec. `sniff`: the directory contains `adventure.json` (a forge workdir also sniffs true — its assembled root document is loadable and importing geometry from a draft is legitimate). `load`: `load_adventure` over the store-read bytes; `title`/`description` populated from the source adventure's `name`/`description`; every level of every dungeon returned as an `ImportedLevel` labeled `"<dungeon-id> level <n>"`. **Load normalizes the payload to what the op vocabulary admits, and every drop or repair lands in `notes`** — a validation-dirty source is a legitimate import source, and a payload the dialog's batch cannot commit would 422 the whole import with no path forward. Concretely: non-canonical and explicit-wall edge entries are dropped; a transition whose *source* cell is out of the level's own bounds is dropped (its target may dangle — that stays, per the op's own rule); of transitions stacked on one source cell, all but the first are dropped (osrlib's own resolution order — the rest are dead data `AddTransition` would rightly refuse); an area's out-of-bounds cells are dropped, and an area losing every cell is dropped whole; a duplicate area id is renamed to the level's next free key, geometry preserved; an out-of-bounds entrance is dropped.

### 7. Type generation

- The new ops ride `AnyEditOp` into `OpBatch` — route-referenced, free. `ImportedGeometry` and the importer route models are route-referenced. `Edge`, `DoorSpec`, `TransitionSpec`, `AreaSpec` already travel inside `Adventure`. The injection list stays exactly `[ApiError]`; the drift gate is unchanged.
- `frontend/src/types/index.ts` grows aliases (`Edge`, `DoorSpec`, `TransitionSpec`, `ImportedGeometry`, the new ops); the vitest type-level suite extends: the union discriminates over the full vocabulary, `SetEdges` values admit `null`, `ImportedLevel.edges` is a `Record`, `Position` stays a two-tuple.

### 8. Frontend map core

Pure logic first, React second — everything in this item is plain TypeScript, unit-tested in node-environment vitest:

- **The edge-key mirror** (`frontend/src/map/edge-key.ts`): `edgeKey(cell, direction)` folding south/east onto the neighbour exactly as osrlib does, plus the parse (split off `x,y` and side, derive the two incident cells — the inline decode at `exploration.py:934-938` is the reference). The mirror is convenience, not authority: the backend op validation is the gate, so a mirror bug is a rejected batch, never a corrupt document.
- **Hit testing**: canvas point → cell, or edge when within a pinned threshold of a cell border (edge targets win inside the threshold; the resolved edge is always canonical). Pure geometry over the view transform.
- **Gesture reducers**: pure `(state, event) -> state` machines per tool, emitting an op-batch description on completion. The compound gestures the spec pins: room rectangle → `CreateArea` (next free key) + `SetEdges` opening every interior edge, one batch; corridor drag → `SetEdges` opening the edges between consecutive path cells; edge paint → the first-clicked assignment applied along the dragged line; area paint/lasso → `SetAreaCells` or `CreateArea`. "Next free key" is the smallest positive integer, as a string, unused as an area id on that level — non-numeric foreign ids simply don't participate.
- **The renderer**: a 2D `<canvas>`, devicePixelRatio-aware, drawing from `(document level, view transform, selection, hover, findings)`. Graph paper per the design language — cream field, fine grid, pencil-weight walls; graphite-on-slate in dark mode — with the classic glyphs the spec names: door ticks on walls, "S" in the gap for secret doors, stair treads, arrows for one-way drops (trapdoor, chute), an entrance marker, area tint with centered key numbers (plain in phase 2 — hollow/filled stocking state is phase 3's), and pencil-palette lint markers (a warning glyph on flagged areas/cells; error findings use the error treatment). A pinned default cell size in CSS pixels, exported as a constant, keeps e2e coordinate math deterministic.
- **View state**: zoom about the cursor (wheel/trackpad), pan (space-drag or middle-drag), fit-level-on-open, and a reset-to-100% control (button and keyboard `0`). In-memory only, per the scope decision. Fit-on-open may land at any zoom, so e2e invokes the reset control before interacting — that is what makes the exported cell-size constant sufficient for coordinate math.

### 9. The map editor surface

The level section of the project screen becomes the map editor; the sidebar tree stays the project-wide navigation and the map chrome adds the spec's own controls, both driving the same selection state:

- **Layout**: toolbar (tool buttons with shortcuts, zoom controls) above the canvas; level tabs per level within the dungeon and the dungeon switcher above the tabs, with add/remove/rename/renumber/resize actions in their menus; a right-side inspector panel for the current selection; the diagnostics panel and header unchanged from phase 1.
- **Tools**: select/inspect (click anything to select and open its inspector; hover shows the cell/edge ref in monospace), room, corridor, wall/door (click cycles wall → open → door-normal; drag paints; a selected door edge opens the door inspector — kind normal/secret, stuck, locked, starts open — committing a `SetEdges`), area (paint into the selected area or create new), entrance (click places; clearing lives in level properties), transition (click source cell → dialog: kind, then target dungeon/level/cell on a mini target-level picker, facing; stairs offer auto-reciprocal creation, default on, landing both `AddTransition`s in one batch; trapdoors and chutes never offer it). Esc cancels a gesture; Delete removes the selection (area with a confirm when it carries content; transition; door/open edge → wall).
- **Commit discipline**: one batch per completed gesture (pointer-up), through the store's existing single-flight queue. Every map commit uses the phase 1 amendment's *builder* form — payloads that depend on current state (the cycle's next value, the next free area key, a repainted cell set) are computed inside the queue against the committed document, so a queued gesture can never resurrect state a previous commit replaced. Builders also emit edge deletes only for keys present in the committed document — painting wall over an already-absent edge is a no-op, never a delete the op would reject — which is what keeps an erase sweep across mixed edges a clean single batch.
- **Management dialogs**: add dungeon (id defaulting to the next free `dungeon-<n>`, name, dimensions defaulting 30×30), add level (number defaulting max+1, dimensions), rename dungeon (cascade stated in the dialog copy), renumber level, resize (current dimensions prefilled; an `op_invariant` rejection renders `details.offenders` as a click-to-navigate list), remove dungeon/level (confirms, disabled on the last one with the reason shown). Level properties popover: dimensions (opening resize), entrance display with clear, and the wandering form relocated from phase 1's `LevelForm` — same ops, new home.
- **Lint on the map**: findings addressed to the visible level render as markers; clicking a diagnostics-panel finding navigates to the level and focuses the target — area selected and outlined, cell ringed, scrolled into view. `navTargetFor` grows `cell`/`edge`/`area` resolution against the document (an address that resolves to nothing stays unnavigable, the phase 1 rule); level-scope findings without a geometry segment (e.g. `wandering_unknown_monster`) open the level properties.

### 10. The import dialog and the torture fixture

- **The flow**: source path → sniff (preselects the importer; one importer means no choice to make) → load → pick a source level (the dialog imports one level per invocation; the payload's multi-level shape is the protocol's, not the dialog's) → destination: a new level in a chosen dungeon (number defaulting next free) or replace an existing level's geometry — the spec's "swap it into the real module". Transitions whose targets won't resolve in the destination document are listed with a drop checkbox, default dropped; kept ones land as authored and surface as validation findings. Importer `notes` render in the dialog.
- **The batch**, one atomic undo step, built in the pinned order that keeps every op valid against the document state it sees: for a new level — `AddLevel`, `SetEdges`, `CreateArea` per area, `AddTransition` per kept transition, `SetEntrance` (when the source has one). For replace — `SetEntrance(None)`, `RemoveTransition` × existing, `RemoveArea` × existing (the dialog warns when content rides along), `SetEdges` deleting every existing key, `ResizeLevel` to the imported dimensions, then the new-level sequence minus `AddLevel`. A `ResizeLevel` rejection (stranded features in a foreign document) rejects the whole import with the offender list — the honest answer until phase 3 can move features.
- **The mapping is pure frontend logic** (`ImportedGeometry` + choices → op list), vitest-covered including both orders; the backend contributes no special apply path.
- **The torture fixture**: an original, editor-buildable two-level geometry exercising every door state (normal, secret, stuck, locked, starts-open), a stairs pair, a one-way chute, a secret-only room, an orphan corridor stub, an overlapping-area pair, and a non-canonical edge key planted in a *doctored* variant only — committed under `tests/fixtures/` with a builder script like phase 0's golden. It serves the lint suite, the round-trip byte-stability suite, the import tests (as a source project), and phase 5's parity test later.
