# Phase 5 plan ÔÇö forge-backed projects

Implementation plan for phase 5 of [the osr-editor spec](spec.md). Phase 5 delivers forge-backed projects: workdir open and detection, the review queue over `report.json` flags with source pages and SVG previews, opÔćĺoverride translation with reasons, the assemble/check loop, stage controls with rerun knobs, and detach ÔÇö plus the sidecar machinery (persisted view state, review marks, per-area author notes) that phases 2 and 3 explicitly deferred here, and the cross-implementation lint-parity test phase 2 pinned for the day `osr-forge` joins the dependencies. Milestone: take a real converted workdir from flagged draft to publishable entirely in the GUI, with `overrides.yaml` as the reviewable record.

Phase 4 (the monster editor) is skipped for now by owner decision; phase 5 lands next and phase 4 follows later. The skip severs nothing here: the one monster surface phase 5 needs ÔÇö forge's printed-notation stat-block correction form ÔÇö consumes forge's `StatBlockOverride` contract, not the native stat-block editor, and the spec itself files it under forge-backed correction ("corrections land pre-mapping by forge's contract and reviewing against the printed page is the point"). What phase 4 still owns ÔÇö native `MonsterTemplate` authoring, clone-and-modify, bundled-template management, the collision rule ÔÇö stays deferred, and when its ops land they will simply join the forge-mode blocked list per this plan's general rule (the spec already names "bundling a monster the module never mentions" a detach-only edit), so no debt accrues.

Six research facts shape this plan more than any others. First, **forge's facade is pinned at exactly five names and will not grow**: `osrforge` re-exports only `convert`, `assemble`, `check`, `estimate`, and `ConversionSettings` (`osrforge/__init__.py:7-13`), forge's own release gate asserts the list verbatim (`osr-forge/docs/phase-4-plan.md:89` ÔÇö "the fa├žade list is the spec's library-API contract and does not grow"), and no promotion task for `rerun` or preview regeneration is recorded anywhere in the forge repo. The editor spec's expectation of a forge-side facade promotion "recorded to land with phase 5" is therefore superseded by forge's own pinned fence: the editor imports `rerun` from `osrforge.convert` (`convert.py:195`) and `render_previews` from `osrforge.assemble` (`assemble.py:1505`) ÔÇö documented, stable module homes ÔÇö with the `>=0.1,<0.2` pin containing the interim risk exactly as the spec provides. Second, **forge's override application is loud, atomic, and fails before any write**: "every entry in a correction file must take effect or application fails with `OverrideError` naming the entry" (`osrforge/overrides.py:6-13`), addressing errors surface "before any `run.json` or artifact write" (`assemble.py:1391-1397`), and duplicate YAML keys are rejected at load by a dedicated loader ("two entries for one key are contradictory corrections", `contracts/overrides.py:262-280`) ÔÇö so the editor's commit protocol can be snapshot Ôćĺ write `overrides.yaml` Ôćĺ `assemble()` Ôćĺ on failure restore the snapshot, with forge's own guarantees ensuring a failed commit never leaves the workdir inconsistent. Third, **the override merge semantics make translation a diff against the assembled document, not against recomputed synthesis**: field semantics are "absent means untouched; explicit `null` means clear" via `model_fields_set` (`contracts/overrides.py:10-13`, honored for `entrance` and `transitions` through explicit `_set` flags, `overrides.py:374-377`), override edges "merge over the synthesized map, override winning per canonical key (a `wall` entry legitimately seals a synthesized opening)" (`overrides.py:411-446`), and the assembled document *is* synthesis-plus-current-overrides ÔÇö so emitting entries for exactly the edges where the desired state differs from the assembled state, merged over the existing entry per canonical key, produces the correct final override map without the editor ever importing `synthesize_geometry`. The spec's "diff against the deterministically recomputed synthesized geometry" is satisfied transitively, and the spec's two residues hold by construction (freed corridor floor stays floor; level dimensions are the bounding box forge recomputes, `overrides.py:446-453`). Fourth, **area overrides embed osrlib's own models post-mapping** (`KeyedEncounter`, `TrapSpec`, `AreaTreasureSpec`, `FeatureSpec` imported directly from `osrlib.crawl.dungeon`, `contracts/overrides.py:132-165`), so phase 3's card payloads translate verbatim; an area *add* must carry `name`, `description`, and a geometry entry supplying its cells, a *remove* rejects any accompanying replacement fields or cells, and the report tombstones both directions (`overridden=("removed",)` / `("added",)`, `assemble.py:1174-1182, 1302`) ÔÇö which is how the translator distinguishes a survey area from an override-added one without ever reading forge's stage caches. Fifth, **there is no `is_workdir()` and no staleness detection, by design**: a workdir is a directory whose `run.json` parses as `RunMeta` (`workdir.py:176`), `assemble()` requires a completed monsters stage and raises `ValueError` otherwise (`assemble.py:1401-1403`), `check()` requires `adventure.json` and `report.json` (`check.py:496-499`), re-assembly "wipes findings by design ÔÇö stale lint about a changed draft is worse than none" (`contracts/report.py:361-364`), and forge's own tests fabricate `run.json` around committed stage caches (`osr-forge/tests/conftest.py:24`) ÔÇö which makes an original, hand-authored stage-cache fixture workdir the right zero-network test vehicle for this phase, with `FixtureProvider` recordings deferred to phase 6 where conversion actually needs them. Sixth, **monster correction keys are normalized names with an exclusivity rule**: both `monsters:` and `monster_templates:` keys match extracted names under `normalize_monster_name` (casefold, collapse whitespace ÔÇö `osrforge/monsters.py:92`), the same name under both kinds is rejected as contradictory (`overrides.py:154-158`), a key matching no extracted name fails loudly listing the cache's unresolved names (`overrides.py:106-112`), and `monster_templates:` entries additionally require the workdir's `statblocks.json` cache state (`assemble.py:1427-1437`) ÔÇö so the monster-resolution panel offers exactly the names the report carries and follows the cache state, and forge's errors are the backstop for anything hand-authored.

## Work items

### 1. Dependencies and the forge bridge

`pyproject.toml` adds `osr-forge>=0.1,<0.2` and `pyyaml>=6` (PyPI, the same posture as `osrlib` ÔÇö the lock resolves published wheels; sibling checkouts remain a local-dev override, not a manifest entry). Imports are pinned per research fact one: the facade names from `osrforge`; `rerun` from `osrforge.convert`; `render_previews` from `osrforge.assemble` (phase 6 consumes it; imported now only if needed ÔÇö nothing in phase 5 calls it); contracts from `osrforge.contracts.run` (`RunMeta`, `Stage`), `.report` (`ExtractionReport`, `parse_flag`), `.overrides` (`Overrides`, `StatBlockOverride`, `MonsterOverride`, `AreaOverride`, `GeometryOverride`, `TownOverride`, `ModuleOverride`, `load_overrides`); `Workdir` from `osrforge.workdir`; `OsrForgeError`/`OverrideError` from `osrforge.errors`; `normalize_monster_name` from `osrforge.monsters`; `canonicalize_edge_key` from `osrforge.overrides`.

- **The materialization seam**: `ProjectStore` grows one method, `materialize(project_id) -> Path` ÔÇö return a local directory holding the project's artifacts, against which file-based tools (osr-forge) may read and write directly. `LocalProjectStore` returns the project directory itself, making materialization the identity and every forge write a live project write. The docstring records the remote-store contract: sync down before returning, re-ingest after forge operations ÔÇö the archive-the-workdir pattern the spec names ÔÇö and that no code outside the bridge may call it (the seam rule).
- **`forge.py`**, the bridge module: `open_workdir_state(path) -> ForgeOpenState` (parse `run.json`, gate, assemble ÔÇö see item 2), `assemble_workdir(path) -> AssembleResult`, `check_workdir(path) -> tuple[LintFinding, ...]`, `rerun_assemble(path, settings_updates) -> ConversionResult` (wrapping `rerun(workdir, Stage.ASSEMBLE, settings_updates=...)` ÔÇö never a model stage in phase 5). All synchronous in-request: assembly is "pure ÔÇŽ no model calls" by forge's contract and `check`'s smoke delve is deterministic and hard-capped (`_COMMAND_CAP = 200`, `check.py:79-82`); no worker thread exists until phase 6's conversion needs one.
- Every `OsrForgeError` crossing the bridge surfaces with forge's own message verbatim ÔÇö forge writes entry-naming, remedy-bearing messages by design and the editor never paraphrases them (item 6 maps the types).

### 2. Forge-backed open and the document service

`open_project`'s forge branch stops refusing and opens. `ProjectTypeUnsupportedError`, its 422 mapping, and its tests are deleted ÔÇö greenfield, no tombstones.

- **The open gates**, in order: `run.json` must parse as `RunMeta` ÔÇö failure is `ForgeWorkdirInvalidError` (422 `forge_workdir_invalid`, remedy naming `run.json` and the detection rule). The monsters stage must be `completed` ÔÇö otherwise `ForgeWorkdirIncompleteError` (422 `forge_workdir_incomplete`, message naming the pending or failed stage from `run.stages`, remedy the CLI `osrforge rerun <stage>` until phase 6 makes resume graphical). Then the bridge runs `assemble()` ÔÇö an `OverrideError` here (a hand-broken `overrides.yaml`) maps to 422 `forge_override_invalid` with forge's entry-naming message; the file is the human-readable record by design and hand-breakage is repaired by hand, while the corrections panel keeps the editor itself from ever producing such a file.
- **Open re-assembles, and that writes.** Workdir artifacts are derived build products forge itself rewrites on every `assemble`; rebuilding on open is the correctness-first posture (the artifacts on disk may be stale against a hand-edited `overrides.yaml`, and forge pins "no automatic staleness detection"). The write is forge's own ÔÇö stage tracking in `run.json`, byte-stable artifacts ÔÇö never the editor's. The open-never-writes invariant is thereby scoped to native projects, where the document is the user's source of truth, and its test says so.
- **Forge `OpenProject` state**: the workdir path (materialized once at open), the current `Overrides` model plus the raw YAML bytes, the assembled `Adventure`, the `ExtractionReport`, the `RunMeta`, the revision counter, and undo/redo stacks of `overrides.yaml` byte snapshots (the spec's rule: the document is derived state). Diagnostics compute exactly as native ÔÇö `compute_diagnostics` over the assembled document ÔÇö plus the forge tier (item 5). The open-time fidelity guard does not apply: the document arrives as forge's in-memory models, not a foreign parse, so `dropped_fields` is `()` for forge projects.
- **The forge commit protocol**, shared by translated op batches (item 3) and override-level edits (item 4): snapshot the current YAML bytes Ôćĺ fold the request into a new `Overrides` value Ôćĺ serialize and write via `store.write_artifact` Ôćĺ `assemble()` Ôćĺ on success bump the revision, adopt the new document/report/overrides, recompute diagnostics, push the snapshot onto the undo stack; on any failure restore the snapshot bytes and re-raise (forge's fail-before-write guarantee means the workdir artifacts never saw the bad file). The batch stays atomic and one undo step; a stale revision is a 409 exactly as native. The delta is the whole-document form (`path=""`) ÔÇö an assembly can move anything, including derived level dimensions ÔÇö and `OpBatchResult` grows an additive `forge: ForgeState | None` carrying the refreshed report/run/overrides (item 6).
- **Undo/redo** pop or push YAML snapshots, write, re-assemble (pure and deterministic, so the restored document is exact), bump the revision, and return the same envelope. The stack depth stays `MAX_UNDO_DEPTH`; snapshots are small text.
- The editor's own persistence path never touches `adventure.json`, `report.json`, or `previews/` in a workdir ÔÇö those bytes are written only inside forge's own functions. This is the reproducibility invariant, and the definition of done proves it behaviorally (a post-session `assemble()` reproduces the session's artifacts byte-for-byte).

### 3. The opÔćĺoverride translator

New module `overrides.py`: `translate_batch(ops, state) -> Overrides` ÔÇö pure, from an applied batch and the current forge state (assembled document, current `Overrides`, report) to the next `Overrides` value. Ops apply to the in-memory document first (the existing `_apply_op` machinery, unchanged ÔÇö the same invariants and rejections hold in both modes), then translate; the assembled result of the new overrides equals the op-applied document, which the test suite asserts on every translation case (the round-trip theorem that keeps the translator honest).

The per-op table:

- `SetAdventureField` Ôćĺ `module:` `name`/`description`/`hooks`. `SetTownField` Ôćĺ `town:` `name`/`description`/`services`/`travel_turns`.
- `SetEncounter`/`SetTrap`/`SetTreasure` Ôćĺ the `areas:` entry's `encounter`/`trap`/`treasure`, the osrlib payload verbatim; `None` becomes explicit `null` (present-and-null clears, per the pinned semantics).
- `AddFeature`/`SetFeature`/`RemoveFeature` at area scope Ôćĺ the `areas:` entry's `features`, the area's full tuple after the batch (forge replaces features wholesale). At level scope Ôćĺ **blocked** (no override kind reaches `LevelSpec.features`).
- `SetAreaField` `name`/`description` Ôćĺ the `areas:` entry; `field="id"` Ôćĺ **blocked** (override addressing is by key; re-keying has no override kind).
- `CreateArea` Ôćĺ an `areas:` add entry (`name`, `description` ÔÇö empty strings satisfy forge's non-null rule) plus a `geometry:` cells entry for the key. Forge-backed mode adds one op invariant: the id must not contain `/` (the address grammar's one reserved character), rejected as `op_invariant`.
- `SetAreaCells` Ôćĺ the `geometry:` entry's `areas[key].cells`. The op already guarantees the key exists on the level, so forge's unknown-key rejection is unreachable from the editor.
- `RemoveArea` Ôćĺ for a survey area, `remove: true` (the fold drops any replacement fields and cells for the address ÔÇö forge rejects the combination); for an override-added area (the report's `"added"` tag), the add entry and its cells are deleted instead ÔÇö forge rejects removing an area the survey lacks.
- `SetEdges` Ôćĺ `geometry:` edge entries under canonical keys: an `Edge` value verbatim; a `None` (the native wall gesture) as an explicit `Edge(kind="wall")` ÔÇö forge's sanctioned seal over a synthesized opening. Entries are emitted for exactly the keys where the desired state differs from the assembled state (research fact three); a delete naming a key the assembled document lacks is the no-op the phase 2 builders already avoid emitting.
- `SetEntrance` Ôćĺ the `geometry:` entry's `entrance` ÔÇö a position, or explicit `null` to clear (supported via `entrance_set`; the resulting `entrance_missing` finding stays the legal, navigable consequence).
- `AddTransition`/`RemoveTransition` Ôćĺ the `geometry:` entry's `transitions`, the level's full tuple after the batch (wholesale replacement; forge drops `transition_guessed` badges with it, which is exactly what correcting a guessed landing should do).
- `SetWandering`, `SetDungeonField`, `RenameDungeon`, `AddDungeon`, `RemoveDungeon`, `AddLevel`, `RemoveLevel`, `RenumberLevel`, `ResizeLevel` Ôćĺ **blocked**: no override kind exists (`WanderingSpec`, dungeon names and ids, level structure) or the value is derived state (dimensions are the bounding box forge recomputes). A blocked op raises `OpUnsupportedForgeError` Ôćĺ 422 `op_unsupported_forge`, `details` naming the op and its target address, remedy the detach offer ÔÇö the spec's "blocks in place with the detach offer," rendered by item 11's dialog. The whole batch rejects (atomicity), before any translation side effect.

The supporting rules:

- **The merge algebra**: the batch folds left-to-right into a net effect per address, then merges over the current `Overrides` ÔÇö one entry per address always (forge's rule), successive edits merging into the entry, a remove superseding prior replacements, a remove-then-create of one survey key collapsing to a replacement. On rewrite, hand-authored east/south edge keys in an existing entry are canonicalized first (`canonicalize_edge_key`) so the merged map can never carry two spellings of one edge ÔÇö the pair forge rejects.
- **Reasons**: every emitted entry carries one. The translator auto-drafts from the ops that touched the entry ÔÇö target-first, change-second, page-anchored when the report has `source_pages` ("area 7 description corrected against p. 14"; "level 1 edges redrawn"; "town services corrected") ÔÇö and successive auto-drafts are latest-wins per the spec. A reason the human has composed in the corrections panel is preserved through later merges: the sidecar tracks which entry keys still carry machine drafts (`auto_reasons`, item 7), and only those are re-drafted ÔÇö the spec's latest-wins governs drafts; a human's composed record is not the editor's to overwrite.
- **Serialization**: pyyaml, block style, kinds in `Overrides` field order, entries sorted by key, plain strings, trailing newline ÔÇö deterministic, git-reviewable, and round-tripped by test through forge's own `load_overrides`. Comments in a pre-existing hand-authored file are not preserved (pyyaml does not round-trip them; the `reason` fields are the record) ÔÇö the same honest normalize-on-first-write posture foreign native documents get, stated in the docs.

### 4. Override-level edits: monster resolution and the corrections panel

Not every correction is an op translation ÔÇö monster remaps and stat-block patches address extracted *names*, and reason editing addresses the overrides file itself. One revision-guarded route carries them: `POST /api/projects/{id}/forge/overrides`, a batch of typed, discriminated edits, applied through the same commit protocol as item 2 (snapshot, write, assemble, refresh ÔÇö one undo step):

- `set_monster_remap {name, template_id, reason?}` / `remove_monster_remap {name}` ÔÇö the name must appear in the report's `unresolved` names or `custom` records (else 422 `request_invalid`; forge's matches-no-extracted-name error remains the backstop for anything stale). The reason defaults to an auto-draft ("remapped to `<template_id>`").
- `set_template_patch {name, patch, reason?}` / `remove_template_patch {name}` ÔÇö `patch` is forge's `StatBlockOverride` shape minus `reason` (printed notation verbatim: AC string and notation, THAC0, hit dice, attack lines, movement, saves, morale, alignment, XP, number appearing, special lines). Cache-state failures (`statblocks.json` absent; `custom_monsters: off`) surface forge's message and remedy verbatim.
- **Exclusivity is resolved at the edit**: committing a remap for a name deletes any template patch for the same normalized name (via `normalize_monster_name`) and vice versa ÔÇö one correction per name, the choice the panel states, never forge's contradictory-corrections error discovered at assemble.
- `set_reason {kind, key, reason}` ÔÇö the corrections panel's inline reason editing; marks the entry human-composed (drops it from `auto_reasons`).
- `remove_entry {kind, key}` ÔÇö per-entry removal at entry grain (a geometry entry removes whole, per level ÔÇö the grain the panel states).

### 5. The forge diagnostics tier, check, and lint parity

- **`Diagnostics` grows `forge: tuple[Finding, ...] = ()`** ÔÇö the additive field the phase 0 envelope reserved, populated by mapping the report's `findings` (`LintFinding`) into the locked shape: `source="forge"` (already in the literal), `code` from the check id, severity and message verbatim, and the `location` grammar (`dungeon/level/area` address, level address, or dungeon id) mapped onto the editor's address grammar so click-to-navigate works unchanged. The delve findings (`delve_blocked` error, `delve_incomplete` warning) arrive only through this tier ÔÇö the editor mirrors forge's static checks, never the delve.
- **Check is on-demand**: `POST /api/projects/{id}/forge/check` runs forge's `check()` (which rewrites `report.json` with findings merged ÔÇö forge's write, through forge's code), refreshes `ForgeState` and the forge tier, and returns the envelope. It is not part of the per-commit loop: re-assembly wipes findings by forge's explicit design ("stale lint about a changed draft is worse than none"), the editor's own live lint already mirrors the five static checks per commit, and the delve is a deliberate, whole-dungeon act ÔÇö so the tier honestly empties on every edit and the pipeline panel's check control shows when no check has run since the last change. Publish keeps its validation-only gate; forge findings join lint in the client-side confirm (item 8).
- **The lint-parity suite** ÔÇö phase 2's pinned deferral, landing now that `osr-forge` is a dependency: for the torture fixture and the small module, fabricate a minimal workdir (the stamped document plus a minimal `ExtractionReport`), run forge's `check()`, and assert the five shared checks agree with `lint_adventure` ÔÇö ids, severities, messages, and locations under the address mapping ÔÇö on both the clean fixtures and their doctored variants. Delve findings are excluded (no editor counterpart); the citations phase 2 pinned stop being the contract and the running comparison becomes it.

### 6. Routes, state, and errors

- **`ProjectState` grows `forge: ForgeState | None = None`** ÔÇö `report: ExtractionReport`, `run: RunMeta`, `overrides: Overrides`, all forge's own pydantic models riding the OpenAPI surface into the generated types (the same source-of-truth discipline as osrlib's models; no hand-written mirror). `OpBatchResult` grows the same optional `forge` field, refreshed on every forge commit, undo, and redo. `ProjectState.sidecar` arrives with item 7.
- **New routes**, all behind the auth dependency: `POST .../forge/overrides` (item 4); `POST .../forge/check` (item 5); `POST .../forge/rerun` with `{settings: dict}` ÔÇö assemble-stage only in phase 5, forge's knobÔćĺowning-stage guard surfaced verbatim as 422 `forge_knob_upstream` when a named knob belongs upstream (the message already carries the remedy), unknown knobs and bad values as `request_invalid` from forge's own settings validation; `POST .../forge/detach` (item 9); `GET .../forge/pages/{n}` serving `pages/{n:04d}.png` bytes through the store (404 `forge_page_not_found` when absent ÔÇö a licensed subset or lean workdir is normal); `GET .../forge/previews/{dungeon_id}/{level_number}` serving the level's SVG the same way.
- **Error vocabulary**: `ForgeWorkdirInvalidError` Ôćĺ 422 `forge_workdir_invalid`; `ForgeWorkdirIncompleteError` Ôćĺ 422 `forge_workdir_incomplete`; `OverrideError` Ôćĺ 422 `forge_override_invalid` (message verbatim ÔÇö forge names the entry); the rerun guard's `ValueError` Ôćĺ 422 `forge_knob_upstream`; `OpUnsupportedForgeError` Ôćĺ 422 `op_unsupported_forge` with `details` `{op, address}` and the detach remedy. `ProjectTypeUnsupportedError` and its mapping are deleted.

### 7. Sidecar machinery: view state, review marks, author notes

`EditorSidecar` grows four additive fields (schema version stays 1, the additive-only fence): `view_state`, `notes`, `review`, `auto_reasons`. Absent fields default empty, so every existing sidecar and every foreign project reads clean. The sidecar is created on first sidecar-bearing write, never at open (open stays read-only for natives, and the forge open's writes are forge's own).

- **`view_state`** ÔÇö active dungeon id and level number, per-level zoom/pan keyed by level address, and the review queue's selected row. Writes coalesce: the frontend flushes on navigation transitions (dungeon or level switch, review-row selection, project close), never per pointer frame ÔÇö the phase 2 concern ("an always-saved editor must not turn every pan into a disk write") answered by the surface that finally earns the writes: resuming correction work across sessions restores where you were.
- **`notes: dict[str, str]`** ÔÇö per-entity author notes keyed by the address grammar, the spec's "per-area author notes live in the sidecar, never the deliverable." The area panel and level properties grow a quiet author-notes field (both project types).
- **The cascade and undo interplay** ÔÇö the design debt phase 3 refused to ship without. Cascading ops (`RenameDungeon`, `RenumberLevel`, `SetAreaField` `id`) re-key affected note, review, and auto-reason entries in the same commit. Each undo-stack entry records the key-remap its op performed; undo replays it inversely and redo forward on the live maps ÔÇö note *content* never rides the stack, so no annotation edit is ever reverted by a document undo. `RemoveArea` leaves its note dormant (unrendered; it reappears if undo restores the area ÔÇö no silent loss). A cascade landing on a key where a dormant note sits overwrites it ÔÇö the live entity's note wins, pinned as acceptable (address-keyed identity is best-effort for dormant addresses). Forge mode needs no cascade: every re-keying op is blocked there.
- **`review`** ÔÇö the review queue's dismissal marks: `{address, flag}` pairs (address `""` for module-scope flags), keyed by the exact flag string so a mark survives re-assembly for as long as the flag it answered does, and goes dormant with the flag when a correction clears it.
- **`auto_reasons`** ÔÇö the kind-qualified entry keys whose reason is still a machine draft (item 3's preserve-human-composition rule).
- **`POST /api/projects/{id}/sidecar`** ÔÇö a typed patch (view state replace; note set/remove by address; review dismiss/undismiss), applied and saved atomically, answering the new sidecar state. Deliberately not revision-guarded: annotation state, single-user, last-write-wins ÔÇö the document's 409 discipline exists to prevent silent *content* destruction, which annotations are not. `ProjectState` grows `sidecar: EditorSidecar`.

### 8. Publish and export for forge-backed projects

Both already operate on "the current document," which forge projects now have; what they need is forge-aware plumbing, not new design:

- **Publish symlink mode** links `adventures/<name>` Ôćĺ the workdir: osr-web discovers `adventures/<dir>/adventure.json`, the workdir has exactly that file at its root, forge's other files ride along as harmlessly as `editor.json` does, and every committed correction republishes live through the always-fresh `adventure.json` ÔÇö the reproducible loop and the live-publish loop compose.
- **Publish copy mode and export copy forge's bytes verbatim** ÔÇö the workdir's `adventure.json` read through the store, never a re-serialization: that file is forge's byte contract (stamped, byte-stable), and the editor's canonical serializer is for documents the editor owns. Native behavior is untouched.
- The gate is unchanged: validation tier empty, server-side; lint *and forge* findings confirm client-side ("secret-only access is sometimes the point" extends naturally to a `delve_incomplete` warning).

### 9. Detach

`POST /api/projects/{id}/forge/detach {path}` ÔÇö the explicit, recorded crossing from the reproducible world to the native one:

- Validates and creates the target like `create_native_project` (same path rules), writes the current assembled `Adventure` through the native canonical serializer, and records provenance in the new sidecar: `SidecarProvenance` grows optional `source_workdir: str | None` and `osrforge_version: str | None` (additive; the version from the workdir's `RunMeta`), alongside the existing `created_by`/`osrlib_version`/`created_at`.
- Notes carry over (addresses are stable across the crossing); review marks and `auto_reasons` stay behind ÔÇö they are forge-review state with no native meaning.
- The workdir is untouched and drops from the open registry; the response is the new native project's `ProjectState` and the frontend navigates into it. Recents list both, each under its own type.
- The confirm dialog states the consequence in the spec's own terms: the forge re-run loop is severed; corrections no longer land in `overrides.yaml`.

### 10. Type generation

`ForgeState` brings `ExtractionReport`, `RunMeta` (with its `ConversionSettings` echo), and `Overrides` (with every override model) into the OpenAPI surface route-referenced ÔÇö generated, never hand-mirrored, exactly as osrlib's models travel. The sidecar, sidecar-patch, override-edit, rerun, and detach request/response models ride the same way. The injection list stays exactly `[ApiError]`; the drift gate is unchanged. `frontend/src/types/index.ts` grows aliases (`ForgeState`, `ExtractionReport`, `AreaReport`, `MonsterSummary`, `CustomMonsterRecord`, `RunMeta`, `Overrides`, `StatBlockOverride`, `EditorSidecar`, the request unions); the vitest type-level suite extends ÔÇö `ProjectState.forge` and `.sidecar` admit their shapes, the override-edit union discriminates, `StatBlockOverride.attacks` is a nullable string array.

### 11. Frontend: the review surfaces

Chrome switches on `ProjectState.type` ÔÇö the map, panels, and diagnostics drawer are shared; forge projects add the review surfaces and the blocked-op behavior:

- **The review queue** ÔÇö a "Review" section in the left nav (forge only): module-scope flag rows first, then per-area rows in report order, each carrying its parsed flags (forge's `flag:detail` grammar split via the contract's own parser), the area's confidence, and its `overridden` badges. Selecting a row navigates ÔÇö level switched, area selected, inspector open ÔÇö and opens the source-pages pane. A dismissal checkbox per row writes the sidecar review marks; a filter hides dismissed rows; the queue header counts undismissed rows, the "walk through the module" affordance the stocking filter established. Validation errors stay in the diagnostics drawer ÔÇö the queue is flags, not a third copy of validation.
- **The source-pages pane** ÔÇö the selected area's `source_pages` rendered from the pages route in a collapsible pane beside the inspector, zoomable, page numbers labeled: correction happens against the printed page.
- **The SVG preview dialog** ÔÇö per level, from the map toolbar in forge mode: forge's own rendering of the corrected plan for eyeballing whole-level geometry beside the editor's canvas.
- **The corrections panel** ÔÇö a nav section listing every override entry grouped by kind in file order: address or name, a one-line summary, the reason inline-editable (`set_reason`), a machine-draft badge until a human composes (auto_reasons), per-entry remove with confirm, click-to-navigate where the entry is addressable. This is the reviewable record, surfaced.
- **The pipeline panel** ÔÇö `run.json` rendered honestly: per-stage status, timestamps, token usage, provider and model identity; the check control with its ran/stale-since-last-change state; the rerun-assemble control offering the assembly-owned knob (`unresolved_fallback`), with the upstream-knob guard's message surfaced verbatim when a hand-entered knob belongs elsewhere.
- **The monster-resolution panel** ÔÇö the report's `monsters` summary as a work surface: unresolved names (with whether a printed block was captured) and custom records (with `derived` badges and source pages). Per name, the two corrections as a stated either/or: remap through the phase 3 monster picker, or the printed-notation form over the `StatBlockOverride` fields ÔÇö visible only when the workdir's cache state supports it, with forge's remedy shown otherwise.
- **The blocked-op dialog** ÔÇö a 422 `op_unsupported_forge` renders in place: the gesture named, the one-line why (this edit has no override kind), and the choice ÔÇö detach (opening the detach dialog) or cancel. No forge-mode surface is preemptively hidden except where this plan says so (import's new-level mode); the spec's rule is "offers it in place when such an edit is attempted."
- **The detach dialog** ÔÇö destination path, the severance warning, the provenance note.
- **Import in forge mode** ÔÇö replace-an-existing-level only (a new level has no override kind; the mode is absent, not disabled). The forge batch variant emits no `ResizeLevel` (dimensions are derived) and otherwise rides the existing mapping; the sealing of every stale synthesized opening falls out of the translator's edge diffing, and the dialog's copy notes the dimension residue (a forge level never shrinks below the synthesized extent).
- **The author-notes card** ÔÇö the sidecar-backed notes field in the area panel and level properties, both project types.
