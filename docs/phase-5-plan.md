# Phase 5 plan — forge-backed projects

Implementation plan for phase 5 of [the osr-editor spec](spec.md). Phase 5 delivers forge-backed projects: workdir open and detection, the review queue over `report.json` flags with source pages and SVG previews, op→override translation with reasons, the assemble/check loop, stage controls with rerun knobs, and detach — plus the sidecar machinery (persisted view state, review marks, per-area author notes) that phases 2 and 3 explicitly deferred here, and the cross-implementation lint-parity test phase 2 pinned for the day `osr-forge` joins the dependencies. Milestone: take a real converted workdir from flagged draft to publishable entirely in the GUI, with `overrides.yaml` as the reviewable record.

Phase 4 (the monster editor) is skipped for now by owner decision; phase 5 lands next and phase 4 follows later. The skip severs nothing here: the one monster surface phase 5 needs — forge's printed-notation stat-block correction form — consumes forge's `StatBlockOverride` contract, not the native stat-block editor, and the spec itself files it under forge-backed correction ("corrections land pre-mapping by forge's contract and reviewing against the printed page is the point"). What phase 4 still owns — native `MonsterTemplate` authoring, clone-and-modify, bundled-template management, the collision rule — stays deferred, and when its ops land they will simply join the forge-mode blocked list per this plan's general rule (the spec already names "bundling a monster the module never mentions" a detach-only edit), so no debt accrues.

Six research facts shape this plan more than any others. First, **forge's facade is pinned at exactly five names and will not grow**: `osrforge` re-exports only `convert`, `assemble`, `check`, `estimate`, and `ConversionSettings` (`osrforge/__init__.py:7-13`), forge pins "the façade list is the spec's library-API contract and does not grow" (`osr-forge/docs/phase-4-plan.md:59`), its release gate asserts the five names verbatim (`phase-4-plan.md:89`), and no promotion task for `rerun` or preview regeneration is recorded anywhere in the forge repo. The editor spec's expectation of a forge-side facade promotion "recorded to land with phase 5" is therefore superseded by forge's own pinned fence: the editor imports `rerun` from `osrforge.convert` (`convert.py:195`) and `render_previews` from `osrforge.assemble` (`assemble.py:1505`) — documented, stable module homes — with the `>=0.1,<0.2` pin containing the interim risk. The spec sentence recording the promotion task is amended alongside this plan (same PR): the source of truth must not keep recording a task that does not exist. Second, **forge's override application is loud, atomic, and fails before any write**: "every entry in a correction file must take effect or application fails with `OverrideError` naming the entry" (`osrforge/overrides.py:6-13`), addressing errors surface "before any `run.json` or artifact write" (`assemble.py:1391-1397`), and duplicate YAML keys are rejected at load by a dedicated loader ("two entries for one key are contradictory corrections", `contracts/overrides.py:262-280`) — so the editor's commit protocol can be snapshot → write `overrides.yaml` → `assemble()` → on failure restore the snapshot, with forge's own guarantees ensuring a failed commit never leaves the workdir inconsistent. Third, **the override merge semantics make translation a diff against the assembled document, not against recomputed synthesis**: field semantics are "absent means untouched; explicit `null` means clear" via `model_fields_set` (`contracts/overrides.py:10-13`, honored for `entrance` and `transitions` through explicit `_set` flags, `overrides.py:374-377`), override edges "merge over the synthesized map, override winning per canonical key (a `wall` entry legitimately seals a synthesized opening)" (`overrides.py:411-446`), and the assembled document *is* synthesis-plus-current-overrides — so emitting entries for exactly the edges where the desired state differs from the assembled state, merged over the existing entry per canonical key, produces the correct final override map without the editor ever importing `synthesize_geometry`. The spec's "diff against the deterministically recomputed synthesized geometry" is satisfied transitively, and the spec's two residues hold by construction (freed corridor floor stays floor; level dimensions are the bounding box forge recomputes, `overrides.py:446-453`). Fourth, **area overrides embed osrlib's own models post-mapping** (`KeyedEncounter`, `TrapSpec`, `AreaTreasureSpec`, `FeatureSpec` imported directly from `osrlib.crawl.dungeon`, `contracts/overrides.py:132-165`), so phase 3's card payloads translate verbatim; an area *add* must carry `name`, `description`, and a geometry entry supplying its cells, a *remove* rejects any accompanying replacement fields or cells, and the report tombstones both directions (`overridden=("removed",)` / `("added",)`, `assemble.py:1174-1182, 1302`) — which is how the translator distinguishes a survey area from an override-added one without ever reading forge's stage caches. Fifth, **there is no `is_workdir()` and no staleness detection, by design**: a workdir is a directory whose `run.json` parses as `RunMeta` (`workdir.py:176`), `assemble()` requires a completed monsters stage and raises `ValueError` otherwise (`assemble.py:1401-1403`), `check()` requires `adventure.json` and `report.json` (`check.py:496-499`), re-assembly "wipes findings by design — stale lint about a changed draft is worse than none" (`contracts/report.py:361-364`), and forge's own tests fabricate `run.json` around committed stage caches (`osr-forge/tests/conftest.py:24`) — which makes an original, hand-authored stage-cache fixture workdir the right zero-network test vehicle for this phase, with `FixtureProvider` recordings deferred to phase 6 where conversion actually needs them. Sixth, **monster correction keys are normalized names with an exclusivity rule**: both `monsters:` and `monster_templates:` keys match extracted names under `normalize_monster_name` (casefold, collapse whitespace — `osrforge/monsters.py:92`), the same name under both kinds is rejected as contradictory (`overrides.py:154-158`), a key matching no extracted name fails loudly listing the cache's unresolved names (`overrides.py:106-112`), and `monster_templates:` entries additionally require the workdir's `statblocks.json` cache state (`assemble.py:1427-1437`) — so the monster-resolution panel offers exactly the names the report carries and follows the cache state, and forge's errors are the backstop for anything hand-authored.

Out of scope (each with the phase that picks it up): conversion end to end — `estimate`, `convert`, provider configuration, progress polling, cancellation, and therefore every rerun whose resumed chain contains a model stage (`preprocess`, `survey`, `content`, `monsters` — forge requires a provider exactly then, `convert.py:260-263`) and the pipeline view for *incomplete* workdirs, which without a provider could show status but action nothing; phase 5 opens assemblable workdirs and answers an incomplete one with a 422 naming the failed stage and the CLI remedy, and phase 6 turns that refusal into the pipeline view with resume (phase 6). The standalone regenerate-previews control is deferred with it: in phase 5's always-assembled world every commit and every open refreshes previews through `assemble()` itself, so a regenerate button would be dead surface (the phase 1 no-dead-surface rule); it ships in phase 6 where pre-assemble workdirs make it actionable. `FixtureProvider`-recorded conversion tests (phase 6, with conversion). The native monster editor and bundled-template ops (phase 4, deferred by the skip — see the intro). Stocking seeds in the sidecar (phase 7, with SRD stocking). Walk mode (phase 8). Docs site and release engineering (phase 9). Remote-store workdir sync-back — the materialization seam documents the contract and the local store satisfies it trivially; a hosted future implements the sync (spec's future extensions).

## Work items

### 1. Dependencies and the forge bridge

`pyproject.toml` adds `osr-forge>=0.1,<0.2` and `pyyaml>=6` (PyPI, the same posture as `osrlib` — the lock resolves published wheels; sibling checkouts remain a local-dev override, not a manifest entry). Imports are pinned per research fact one: the facade names from `osrforge`; `rerun` from `osrforge.convert`; `render_previews` from `osrforge.assemble` (phase 6 consumes it; imported now only if needed — nothing in phase 5 calls it); contracts from `osrforge.contracts.run` (`RunMeta`, `Stage`), `.report` (`ExtractionReport`, `parse_flag`), `.overrides` (`Overrides`, `StatBlockOverride`, `MonsterOverride`, `AreaOverride`, `GeometryOverride`, `TownOverride`, `ModuleOverride`, `load_overrides`); `Workdir` from `osrforge.workdir`; `OsrForgeError`/`OverrideError` from `osrforge.errors`; `normalize_monster_name` from `osrforge.monsters`; `canonicalize_edge_key` from `osrforge.overrides`.

- **The materialization seam**: `ProjectStore` grows one method, `materialize(project_id) -> Path` — return a local directory holding the project's artifacts, against which file-based tools (osr-forge) may read and write directly. `LocalProjectStore` returns the project directory itself, making materialization the identity and every forge write a live project write. The docstring records the remote-store contract: sync down before returning, re-ingest after forge operations — the archive-the-workdir pattern the spec names — and that no code outside the bridge may call it (the seam rule).
- **`forge.py`**, the bridge module: `open_workdir_state(path) -> ForgeOpenState` (parse `run.json`, gate, assemble — see item 2), `assemble_workdir(path) -> AssembleResult`, `check_workdir(path) -> tuple[LintFinding, ...]`, `rerun_assemble(path, settings_updates) -> ConversionResult` (wrapping `rerun(workdir, Stage.ASSEMBLE, settings_updates=...)` — never a model stage in phase 5). All synchronous in-request: assembly is "pure … no model calls" by forge's contract and `check`'s smoke delve is deterministic and hard-capped (`_COMMAND_CAP = 200`, `check.py:79-82`); no worker thread exists until phase 6's conversion needs one.
- Every `OsrForgeError` crossing the bridge surfaces with forge's own message verbatim — forge writes entry-naming, remedy-bearing messages by design and the editor never paraphrases them (item 6 maps the types).

### 2. Forge-backed open and the document service

`open_project`'s forge branch stops refusing and opens. `ProjectTypeUnsupportedError`, its 422 mapping, and its tests are deleted — greenfield, no tombstones.

- **The open gates**, in order: `run.json` must parse as `RunMeta` — failure is `ForgeWorkdirInvalidError` (422 `forge_workdir_invalid`, remedy naming `run.json` and the detection rule). The monsters stage must be `completed` — otherwise `ForgeWorkdirIncompleteError` (422 `forge_workdir_incomplete`, message naming the pending or failed stage from `run.stages`, remedy the CLI `osrforge rerun <stage>` until phase 6 makes resume graphical). Then the bridge runs `assemble()` — a broken `overrides.yaml` maps to 422 `forge_override_invalid` whatever the break: `OverrideError` (an entry that cannot take effect — forge's message names it), the contract's `pydantic.ValidationError`, or `yaml.YAMLError`, the full failure set `load_overrides` raises (`contracts/overrides.py:296-299`) and the set forge's own CLI catches. Any other `ValueError` out of `assemble()` (a missing or stale stage cache) maps to 422 `forge_workdir_invalid`, message verbatim. The overrides file is the human-readable record by design and hand-breakage is repaired by hand, while the corrections panel keeps the editor itself from ever producing such a file.
- **Open re-assembles, and that writes.** Workdir artifacts are derived build products forge itself rewrites on every `assemble`; rebuilding on open is the correctness-first posture (the artifacts on disk may be stale against a hand-edited `overrides.yaml`, and forge pins "no automatic staleness detection"). The write is forge's own — stage tracking in `run.json`, byte-stable artifacts — never the editor's. The open-never-writes invariant is thereby scoped to native projects, where the document is the user's source of truth, and its test says so.
- **Forge `OpenProject` state**: the workdir path (materialized once at open), the current `Overrides` model plus the raw YAML bytes, the assembled `Adventure`, the `ExtractionReport`, the `RunMeta`, the revision counter, and undo/redo stacks of **snapshot pairs** — the `overrides.yaml` bytes and the `auto_reasons` set together (the spec's rule: the document is derived state, and `auto_reasons` is derived state of the same commit — item 7). Diagnostics compute exactly as native — `compute_diagnostics` over the assembled document — plus the forge tier (item 5). The open-time fidelity guard does not apply: the document arrives as forge's in-memory models, not a foreign parse, so `dropped_fields` is `()` for forge projects.
- **The forge commit protocol**, shared by translated op batches (item 3) and override-level edits (item 4): snapshot the current YAML bytes and `auto_reasons` → fold the request into a new `Overrides` value (and its `auto_reasons` delta) → serialize and write the YAML via `store.write_artifact`, persist the sidecar's `auto_reasons` → `assemble()` → on success bump the revision, adopt the new document/report/overrides, recompute diagnostics, push the snapshot pair onto the undo stack; on any failure restore the snapshotted bytes and `auto_reasons` and re-raise (forge's fail-before-write guarantee means the workdir artifacts never saw the bad file). The batch stays atomic and one undo step; a stale revision is a 409 exactly as native. The delta is the whole-document form (`path=""`) — an assembly can move anything, including derived level dimensions — and `OpBatchResult` grows an additive `forge: ForgeState | None` carrying the refreshed report/run/overrides (item 6).
- **Undo/redo** pop or push snapshot pairs, write the YAML and `auto_reasons`, re-assemble (pure and deterministic, so the restored document is exact), bump the revision, and return the same envelope — the reason text and its auto/human flag always restored together. The stack depth stays `MAX_UNDO_DEPTH`; snapshots are small text.
- The editor's own persistence path never touches `adventure.json`, `report.json`, or `previews/` in a workdir — those bytes are written only inside forge's own functions. This is the reproducibility invariant, and the definition of done proves it behaviorally (a post-session `assemble()` reproduces the session's artifacts byte-for-byte).

### 3. The op→override translator

New module `overrides.py`: `translate_batch(ops, state) -> Overrides` — pure, from an applied batch and the current forge state (assembled document, current `Overrides`, report) to the next `Overrides` value. Ops apply to the in-memory document first through the existing `_apply_op` machinery — the same invariants and rejections hold in both modes, with one pinned substitution (the growth rule below) — then translate. The round-trip theorem that keeps the translator honest: the assembled result of the new overrides equals the op-applied document **under a derivation-aware equivalence**, asserted by the suite on every translation case — equal modulo edge representation (an absent entry and an explicit `Edge(kind="wall")` are the same wall: the native model expresses by absence what forge's merge stores as a seal), modulo level `width`/`height` (derived state forge recomputes as the final plan's bounding box, `overrides.py:450-454`), and modulo `Adventure.monsters` (assembly bundles only the custom templates the built encounters still reference, `assemble.py:1318-1331` — a `SetEncounter` dropping the last reference to an emitted template drops the template). Everything else — areas and their order, content, edges semantically, entrance, transitions, town and module metadata — must match exactly. This equivalence is also why the editor adopts the *assembled* document after every commit rather than trusting the candidate.

The per-op table:

- `SetAdventureField` → `module:` `name`/`description`/`hooks`. `SetTownField` → `town:` `name`/`description`/`services`/`travel_turns`.
- `SetEncounter`/`SetTrap`/`SetTreasure` → the `areas:` entry's `encounter`/`trap`/`treasure`, the osrlib payload verbatim; `None` becomes explicit `null` (present-and-null clears, per the pinned semantics).
- `AddFeature`/`SetFeature`/`RemoveFeature` at area scope → the `areas:` entry's `features`, the area's full tuple after the batch (forge replaces features wholesale). At level scope → **blocked** (no override kind reaches `LevelSpec.features`).
- `SetAreaField` `name`/`description` → the `areas:` entry; `field="id"` → **blocked** (override addressing is by key; re-keying has no override kind).
- `CreateArea` → an `areas:` add entry (`name`, `description` — empty strings satisfy forge's non-null rule) plus a `geometry:` cells entry for the key. Forge-backed mode adds one op invariant: the id must not contain `/` (the address grammar's one reserved character), rejected as `op_invariant`.
- `SetAreaCells` → the `geometry:` entry's `areas[key].cells`. The op already guarantees the key exists on the level, so forge's unknown-key rejection is unreachable from the editor.
- `RemoveArea` → for a survey area, `remove: true` (the fold drops any replacement fields and cells for the address — forge rejects the combination); for an override-added area (the report's `"added"` tag), the add entry and its cells are deleted instead — forge rejects removing an area the survey lacks.
- `SetEdges` → `geometry:` edge entries under canonical keys: an `Edge` value verbatim; a `None` (the native wall gesture) as an explicit `Edge(kind="wall")` — forge's sanctioned seal over a synthesized opening. The diff is **semantic**: an absent entry and an explicit wall entry are the same wall, so entries are emitted for exactly the keys whose desired state differs *in effect* from the assembled state (research fact three). Two consequences: a delete naming an edge the assembled document already treats as wall is a no-op, and the wall gesture over an explicit wall entry (which an assembled document legally carries after any merge) keeps the existing override entry rather than removing it — dropping the entry could silently re-open the synthesized passage underneath it, which only synthesis could reveal.
- `SetEntrance` → the `geometry:` entry's `entrance` — a position, or explicit `null` to clear (supported via `entrance_set`; the resulting `entrance_missing` finding stays the legal, navigable consequence).
- `AddTransition`/`RemoveTransition` → the `geometry:` entry's `transitions`, the level's full tuple after the batch (wholesale replacement; forge drops `transition_guessed` badges with it, which is exactly what correcting a guessed landing should do).
- `SetWandering`, `SetDungeonField`, `RenameDungeon`, `AddDungeon`, `RemoveDungeon`, `AddLevel`, `RemoveLevel`, `RenumberLevel`, `ResizeLevel` → **blocked**: no override kind exists (`WanderingSpec`, dungeon names and ids, level structure) or the value is derived state (dimensions are the bounding box forge recomputes). A blocked op raises `OpUnsupportedForgeError` → 422 `op_unsupported_forge`, `details` naming the op and its target address, remedy the detach offer — the spec's "blocks in place with the detach offer," rendered by item 11's dialog. The whole batch rejects (atomicity), before any translation side effect.

The supporting rules:

- **The growth rule** — forge-mode geometry admits growth beyond the current extent. The spec's flagship case ("redrawing rooms, walls, doors ... — or importing a level drawn in a scratch project — writes `geometry:` entries, no detach required") requires drawing past the current dimensions, because a forge level's dimensions are the derived bounding box and `ResizeLevel` is blocked — under the native bounds invariants, forge-mode geometry could never grow, an unrecorded spec violation. So in forge mode the geometry ops' in-bounds invariants (`CreateArea` and `SetAreaCells` cells, `SetEdges` incident cells, `SetEntrance`, `AddTransition` sources) become **nonnegative** invariants — forge's own guard, and the only one it has (`overrides.py:254-257`). The candidate document legally carries cells beyond its stale `width`/`height` for the instant before the assembled document's recomputed bounding box is adopted (out-of-bounds cells are validation findings, not model errors, so the candidate stays constructible), and shrinking keeps the documented residue: dimensions never fall below the synthesized floor plan's extent.
- **The merge algebra**: the batch folds left-to-right into a net effect per address, then merges over the current `Overrides` — one entry per address always (forge's rule), successive edits merging into the entry, a remove superseding prior replacements, a remove-then-create of one survey key collapsing to a replacement. A collapse — and any `CreateArea` over a survey key an earlier commit removed — additionally sets explicit `null` for `encounter`, `trap`, `treasure`, and `features`: a replacement entry leaves unset survey fields in force (`_build_area` reads overridden fields only, `assemble.py:1006-1057`), and a recreated area must not resurrect the survey's content. On rewrite, hand-authored east/south edge keys in an existing entry are canonicalized first (`canonicalize_edge_key`) so the merged map can never carry two spellings of one edge — the pair forge rejects.
- **Reasons**: every emitted entry carries one. The translator auto-drafts from the ops that touched the entry — target-first, change-second, page-anchored when the report has `source_pages` ("area 7 description corrected against p. 14"; "level 1 edges redrawn"; "town services corrected") — and successive auto-drafts are latest-wins per the spec. A reason the human has composed in the corrections panel is preserved through later merges: the sidecar tracks which entry keys still carry machine drafts (`auto_reasons`, item 7), and only those are re-drafted — the spec's latest-wins governs drafts; a human's composed record is not the editor's to overwrite.
- **Serialization**: pyyaml with `sort_keys=False`, block style, kinds in `Overrides` field order, plain strings, trailing newline — and **insertion-preserving entry order**: existing entries keep their file order, new entries append, never a sort. Entry order is semantic in `areas:` — adds land in the draft in correction-file order, "the one ordering rule `build_draft` and `render_previews` must agree on" (`osrforge/overrides.py:470-489`, `assemble.py:1294-1304`) — so sorting would reorder a hand-authored file's draft and scramble the editor's own creation order; insertion order is deterministic and becomes the uniform rule for every kind. Round-tripped by test through forge's own `load_overrides`. Comments in a pre-existing hand-authored file are not preserved (pyyaml does not round-trip them; the `reason` fields are the record) — the same honest normalize-on-first-write posture foreign native documents get, stated in the docs.

### 4. Override-level edits: monster resolution and the corrections panel

Not every correction is an op translation — monster remaps and stat-block patches address extracted *names*, and reason editing addresses the overrides file itself. One revision-guarded route carries them: `POST /api/projects/{id}/forge/overrides`, a batch of typed, discriminated edits, applied through the same commit protocol as item 2 (snapshot, write, assemble, refresh — one undo step):

- `set_monster_remap {name, template_id, reason?}` / `remove_monster_remap {name}` — the route accepts any name: forge's cache-membership check is the honest backstop (a name matching no extracted name fails the commit loudly, listing the cache's unresolved names, surfaced verbatim), and forge *accepts* a remap on a resolved name — its documented remedy for a flagless wrong pick ("an entry on a name the tiers resolved forces emission", `contracts/overrides.py:108-110`). The panel offers the names the report carries — `unresolved` plus the `custom` records; the report lists no resolved names, so the resolved-name remap stays reachable by hand-authoring (listed, reasoned, and editable like every entry) while the GUI's first-class correction for a wrong resolved pick remains replacing the encounter — the narrowing recorded here. The reason defaults to an auto-draft ("remapped to `<template_id>`").
- `set_template_patch {name, patch, reason?}` / `remove_template_patch {name}` — `patch` is forge's `StatBlockOverride` shape minus `reason` (printed notation verbatim: AC string and notation, THAC0, hit dice, class and level, fixed hp, attack lines, movement, saves, morale, alignment, XP, number appearing, special lines). Cache-state failures (`statblocks.json` absent; `custom_monsters: off`) surface forge's message and remedy verbatim.
- **Exclusivity is resolved at the edit**: committing a remap for a name deletes any template patch for the same normalized name (via `normalize_monster_name`) and vice versa — one correction per name, the choice the panel states, never forge's contradictory-corrections error discovered at assemble.
- `set_reason {kind, key, reason}` — the corrections panel's inline reason editing; marks the entry human-composed (drops it from `auto_reasons`).
- `remove_entry {kind, key}` — per-entry removal at entry grain (a geometry entry removes whole, per level — the grain the panel states).

### 5. The forge diagnostics tier, check, and lint parity

- **`Diagnostics` grows `forge: tuple[Finding, ...] = ()`** — the additive field the phase 0 envelope reserved, populated by mapping the report's `findings` (`LintFinding`) into the locked shape: `source="forge"` (already in the literal), `code` from the check id, severity and message verbatim, and the `location` grammar (`dungeon/level/area` address, level address, or dungeon id) mapped onto the editor's address grammar so click-to-navigate works unchanged. The delve findings (`delve_blocked` error, `delve_incomplete` warning) arrive only through this tier — the editor mirrors forge's static checks, never the delve.
- **Check is on-demand**: `POST /api/projects/{id}/forge/check` runs forge's `check()` (which rewrites `report.json` with findings merged — forge's write, through forge's code), refreshes `ForgeState` and the forge tier, and returns the envelope. It is not part of the per-commit loop: re-assembly wipes findings by forge's explicit design ("stale lint about a changed draft is worse than none"), the editor's own live lint already mirrors the five static checks per commit, and the delve is a deliberate, whole-dungeon act — so the tier honestly empties on every edit and the pipeline panel's check control shows when no check has run since the last change. Publish keeps its validation-only gate; forge findings join lint in the client-side confirm (item 8).
- **The lint-parity suite** — phase 2's pinned deferral, landing now that `osr-forge` is a dependency: for the torture fixture and the small module, fabricate a minimal workdir (the stamped document plus a minimal `ExtractionReport`), run forge's `check()`, and assert the five shared checks agree with `lint_adventure` — ids, severities, messages, and locations under the address mapping — on both the clean fixtures and their doctored variants. Delve findings are excluded (no editor counterpart); the citations phase 2 pinned stop being the contract and the running comparison becomes it.

### 6. Routes, state, and errors

- **`ProjectState` grows `forge: ForgeState | None = None`** — `report: ExtractionReport`, `run: RunMeta`, `overrides: Overrides`, all forge's own pydantic models riding the OpenAPI surface into the generated types (the same source-of-truth discipline as osrlib's models; no hand-written mirror). `OpBatchResult` grows the same optional `forge` field, refreshed on every forge commit, undo, and redo. `ProjectState.sidecar` arrives with item 7.
- **New routes**, all behind the auth dependency: `POST .../forge/overrides` (item 4); `POST .../forge/check` (item 5); `POST .../forge/rerun` with `{settings: dict}` — assemble-stage only in phase 5. `rerun` raises `ValueError` for more than the knob→owning-stage drift guard (stage preconditions, missing caches), and the editor never discriminates forge's errors by message-sniffing: every `ValueError` from `rerun` maps to one code, 422 `forge_rerun_invalid`, message verbatim — the guard's message already carries its remedy ("rerun `<owning-stage>` instead"). Unknown knobs and bad values are `request_invalid` from forge's own settings validation; `POST .../forge/detach` (item 9); `GET .../forge/pages/{n}` serving `pages/{n:04d}.png` bytes through the store (404 `forge_page_not_found` when absent — a licensed subset or lean workdir is normal); `GET .../forge/previews/{dungeon_id}/{level_number}` serving the level's SVG the same way.
- **Error vocabulary**: `ForgeWorkdirInvalidError` → 422 `forge_workdir_invalid` (unparseable `run.json`; `assemble()`'s cache-state `ValueError`s); `ForgeWorkdirIncompleteError` → 422 `forge_workdir_incomplete`; the `load_overrides` failure set (`OverrideError`, the contract `ValidationError`, `yaml.YAMLError`) and every applying `OverrideError` → 422 `forge_override_invalid` (message verbatim — forge names the entry); `rerun`'s `ValueError`s → 422 `forge_rerun_invalid`; `OpUnsupportedForgeError` → 422 `op_unsupported_forge` with `details` `{op, address}` and the detach remedy. `ProjectTypeUnsupportedError` and its mapping are deleted.

### 7. Sidecar machinery: view state, review marks, author notes

`EditorSidecar` grows four additive fields (schema version stays 1, the additive-only fence): `view_state`, `notes`, `review`, `auto_reasons`. Absent fields default empty, so every existing sidecar and every foreign project reads clean. `SidecarProvenance` becomes `provenance: SidecarProvenance | None` — an additive relaxation (a required field made optional never breaks an existing reader), because a foreign project the editor merely opens has no provenance to claim, and `ProjectState.sidecar` must be constructible for it: the server always answers a sidecar, an in-memory `EditorSidecar()` (provenance `None`, every field empty) for a project with none on disk. The sidecar file is written on first sidecar-bearing write, never at open (open stays read-only for natives, and the forge open's writes are forge's own); a foreign project's first note thus persists a sidecar with `provenance=None`, honest about what the editor did and didn't author.

- **`view_state`** — active dungeon id and level number, per-level zoom/pan keyed by level address, and the review queue's selected row. Writes coalesce: the frontend flushes on navigation transitions (dungeon or level switch, review-row selection, project close), never per pointer frame — the phase 2 concern ("an always-saved editor must not turn every pan into a disk write") answered by the surface that finally earns the writes: resuming correction work across sessions restores where you were.
- **`notes: dict[str, str]`** — per-entity author notes keyed by the address grammar, the spec's "per-area author notes live in the sidecar, never the deliverable." The area panel and level properties grow a quiet author-notes field (both project types).
- **The note cascade and undo interplay** — the design debt phase 3 refused to ship without. `notes` is the only sidecar field the cascade touches: the re-keying ops (`RenameDungeon`, `RenumberLevel`, `SetAreaField` `id`) are all native-mode ops (forge blocks every one), `notes` exists in both modes, and `review`/`auto_reasons` are forge-only — so they never coincide with a re-key and need no cascade. A cascading op re-keys affected note entries in the same commit; each undo-stack entry records the note key-remap its op performed, and undo replays it inversely, redo forward, on the live map — note *content* never rides the document stack, so no annotation edit is ever reverted by a document undo. `RemoveArea` leaves its note dormant (unrendered; it reappears if undo restores the area — no silent loss). A cascade landing on a key where a dormant note sits overwrites it — the live entity's note wins, pinned as acceptable (address-keyed identity is best-effort for dormant addresses).
- **`review`** — the review queue's dismissal marks: `{address, flag}` pairs (address `""` for module-scope flags), keyed by the exact flag string so a mark survives re-assembly for as long as the flag it answered does, and goes dormant with the flag when a correction clears it. Forge-only; the report's flags have no native counterpart.
- **`auto_reasons`** — the kind-qualified override-entry keys whose reason is still a machine draft (item 3's preserve-human-composition rule). Forge-only, and the one sidecar field a *document* commit changes rather than the sidecar route: the translator drops an entry from `auto_reasons` the moment a human composes its reason (`set_reason`, item 4), and a translated batch adds an entry when it drafts a fresh reason. So `auto_reasons` is derived state of the same forge commit that writes `overrides.yaml`, and it rides the forge undo stack *with* the YAML snapshot (item 2's pinned pair) — undo restores the reason text and its auto/human flag together, never the drift of one restored without the other.
- **`POST /api/projects/{id}/sidecar`** — a typed patch (view state replace; note set/remove by address; review dismiss/undismiss), applied and saved atomically, answering the new sidecar state. Deliberately not revision-guarded: annotation state, single-user, last-write-wins — the document's 409 discipline exists to prevent silent *content* destruction, which annotations are not. `ProjectState` grows `sidecar: EditorSidecar`.

### 8. Publish and export for forge-backed projects

Both already operate on "the current document," which forge projects now have; what they need is forge-aware plumbing, not new design:

- **Publish symlink mode** links `adventures/<name>` → the workdir: osr-web discovers `adventures/<dir>/adventure.json`, the workdir has exactly that file at its root, forge's other files ride along as harmlessly as `editor.json` does, and every committed correction republishes live through the always-fresh `adventure.json` — the reproducible loop and the live-publish loop compose.
- **Publish copy mode and export copy forge's bytes verbatim** — the workdir's `adventure.json` read through the store, never a re-serialization: that file is forge's byte contract (stamped, byte-stable), and the editor's canonical serializer is for documents the editor owns. Native behavior is untouched.
- The gate is unchanged: validation tier empty, server-side; lint *and forge* findings confirm client-side ("secret-only access is sometimes the point" extends naturally to a `delve_incomplete` warning).

### 9. Detach

`POST /api/projects/{id}/forge/detach {path}` — the explicit, recorded crossing from the reproducible world to the native one:

- Validates and creates the target like `create_native_project` (same path rules), writes the current assembled `Adventure` through the native canonical serializer, and records provenance in the new sidecar: `SidecarProvenance` grows optional `source_workdir: str | None` and `osrforge_version: str | None` (additive; the version from the workdir's `RunMeta`), alongside the existing `created_by`/`osrlib_version`/`created_at`.
- Notes carry over (addresses are stable across the crossing); review marks and `auto_reasons` stay behind — they are forge-review state with no native meaning.
- The workdir is untouched and drops from the open registry; the response is the new native project's `ProjectState` and the frontend navigates into it. Recents list both, each under its own type.
- The confirm dialog states the consequence in the spec's own terms: the forge re-run loop is severed; corrections no longer land in `overrides.yaml`.

### 10. Type generation

`ForgeState` brings `ExtractionReport`, `RunMeta` (with its `ConversionSettings` echo), and `Overrides` (with every override model) into the OpenAPI surface route-referenced — generated, never hand-mirrored, exactly as osrlib's models travel. The sidecar, sidecar-patch, override-edit, rerun, and detach request/response models ride the same way. The injection list stays exactly `[ApiError]`; the drift gate is unchanged. `frontend/src/types/index.ts` grows aliases (`ForgeState`, `ExtractionReport`, `AreaReport`, `MonsterSummary`, `CustomMonsterRecord`, `RunMeta`, `Overrides`, `StatBlockOverride`, `EditorSidecar`, the request unions); the vitest type-level suite extends — `ProjectState.forge` and `.sidecar` admit their shapes, the override-edit union discriminates, `StatBlockOverride.attacks` is a nullable string array.

### 11. Frontend: the review surfaces

Chrome switches on `ProjectState.type` — the map, panels, and diagnostics drawer are shared; forge projects add the review surfaces and the blocked-op behavior:

- **The review queue** — a "Review" section in the left nav (forge only). The queue is a work list of *flags*, so its rows are the areas (and the module) that carry at least one flag: a flagless area gets no row (it is not review work), module-scope flags lead as a row at address `""`, then per-area rows in report order. Each row is the navigation unit — its area's confidence, its `overridden` badges, and its individual parsed flags (forge's `flag:detail` grammar split via the contract's own parser) — and selecting it navigates (level switched, area selected, inspector open) and opens the source-pages pane. **Dismissal is per flag**, matching the `{address, flag}` mark grain: each listed flag carries its own checkbox writing one mark, and a row-level "dismiss all" is convenience over those marks. A row counts as dismissed when every flag it carries is marked; the filter hides fully-dismissed rows; the header counts **undismissed flags** — the honest work-remaining number, the "walk through the module" affordance the stocking filter established. Validation errors stay in the diagnostics drawer — the queue is flags, not a third copy of validation.
- **The source-pages pane** — the selected area's `source_pages` rendered from the pages route in a collapsible pane beside the inspector, zoomable, page numbers labeled: correction happens against the printed page.
- **The SVG preview dialog** — per level, from the map toolbar in forge mode: forge's own rendering of the corrected plan for eyeballing whole-level geometry beside the editor's canvas.
- **The corrections panel** — a nav section listing every override entry grouped by kind in file order: address or name, a one-line summary, the reason inline-editable (`set_reason`), a machine-draft badge until a human composes (auto_reasons), per-entry remove with confirm, click-to-navigate where the entry is addressable. This is the reviewable record, surfaced.
- **The pipeline panel** — `run.json` rendered honestly: per-stage status, timestamps, token usage, provider and model identity; the check control with its ran/stale-since-last-change state; the rerun-assemble control offering the assembly-owned knob (`unresolved_fallback`), with the upstream-knob guard's message surfaced verbatim when a hand-entered knob belongs elsewhere.
- **The monster-resolution panel** — the report's `monsters` summary as a work surface: unresolved names (with whether a printed block was captured) and custom records (with `derived` badges and source pages). Per name, the two corrections as a stated either/or: remap through the phase 3 monster picker, or the printed-notation form over the `StatBlockOverride` fields — visible only when the workdir's cache state supports it, with forge's remedy shown otherwise.
- **The blocked-op dialog** — a 422 `op_unsupported_forge` renders in place: the gesture named, the one-line why (this edit has no override kind), and the choice — detach (opening the detach dialog) or cancel. No forge-mode surface is preemptively hidden except where this plan says so (import's new-level mode); the spec's rule is "offers it in place when such an edit is attempted."
- **The detach dialog** — destination path, the severance warning, the provenance note.
- **Import in forge mode** — replace-an-existing-level only (a new level has no override kind; the mode is absent, not disabled). The forge batch variant emits no `ResizeLevel` (dimensions are derived) and otherwise rides the existing mapping; the sealing of every stale synthesized opening falls out of the translator's edge diffing, and the dialog's copy notes the dimension residue (a forge level never shrinks below the synthesized extent).
- **The author-notes card** — the sidecar-backed notes field in the area panel and level properties, both project types.

### 12. The forge-workdir fixture

`tests/fixtures/forge_workdir/` plus `tests/generate_forge_workdir.py`, the established generator pattern: an original mini-module authored directly as forge stage caches through forge's own contract models — `run.json` fabricated (forge's own test pattern) with the monsters stage completed and a settings echo with `custom_monsters: emit`; `stages/survey.json` describing one dungeon of two levels; the per-level area caches; `monsters.json` with one exact-resolving name and two unresolved; `statblocks.json` carrying a usable printed block for one of them (emitting a custom template with a `derived` list) and, for the other, the contract's explicit `null` block marker — the `monster_unresolved` stand-in — **not** an absent entry: under `custom_monsters: emit` a cached-unresolved name with no `blocks` entry is a stale cache that hard-errors in `assemble()` (`assemble.py:1418-1424`), so the marker must be the explicit null the contract defines (`contracts/stages.py:513-517`); two tiny committed page PNGs with text layers. The content is built to raise the review vocabulary: `geometry_synthesized`, `monster_custom`, `monster_unresolved`, `low_confidence`, `connection_ambiguous`, `transition_guessed`, and `treasure_unparsed` (an unparseable treasure string), and to reach validation-clean through a scripted set of corrections the e2e milestone applies. Tests copy the fixture to a temp directory before any forge operation — assembly writes artifacts, and the checkout stays clean. Wholly original content, per the licensing fence; `FixtureProvider` recordings wait for phase 6, which is the first phase that runs a model stage.

- **bridge and open**: the three open gates (unparseable `run.json`, incomplete monsters stage with the stage named, broken overrides with forge's message passed through); assemble-on-open freshness (a hand-edited `overrides.yaml` is reflected at open); the open-never-writes test rescoped to native; `materialize` protocol conformance.
- **translator**: per-op golden `overrides.yaml` outputs — the spec's op→override translation goldens — across the whole supported table; the round-trip theorem on every case (assemble of the translated overrides equals the op-applied document); the merge algebra (successive edits one entry; remove drops fields and cells; removing an added area deletes the add; remove-then-create collapses to replace; hand-authored east/south keys canonicalized; explicit wall seals; transitions wholesale; entrance set and cleared); the blocked-op matrix, whole-batch rejection, and the `details` shape; the forge-mode `/` id invariant; reason drafting (page-anchored, latest-wins across drafts, human-composed preserved via `auto_reasons`); serialization determinism and the `load_overrides` round-trip.
- **forge commits**: the snapshot protocol (failure restores bytes, revision unchanged, artifacts untouched); undo/redo as snapshot restore with exact re-derivation; 409 on stale revision; `OpBatchResult.forge` refresh.
- **override-level edits**: remap and patch application; the exclusivity swap; unknown-name rejection; cache-state errors surfaced; reason edit and entry removal; one undo step each.

- **sidecar**: additive defaults over existing sidecars; the patch route; the cascade with undo/redo remap replay (rename, renumber, re-key), dormant notes through remove-and-undo, the dormant-collision rule; review-mark persistence across re-assembly.
- **publish and export**: forge symlink resolving to the workdir and republishing live after a correction; copy and export bytes equal to the workdir's `adventure.json` verbatim; the unchanged gate.

- **frontend (vitest)**: flag-grammar parsing; review-row building, dismissal filtering, and counts; corrections-panel grouping and reason editing; the printed-notation form builders; pipeline-panel state incl. the stale-check hint; the blocked-op dialog flow into detach; the forge import-mapping variant; the notes card; type-level assertions.

## Definition of done

- The full gauntlet stays green on both toolchains, extended by the new suites; the typegen drift gate passes with the injection list still `[ApiError]`.
- The milestone works end to end by hand and in Playwright: a flagged workdir taken to publishable entirely in the GUI — review queue to corrected areas to resolved monsters to a clean validation tier to a live symlink publish — with `overrides.yaml` as the reviewable record, and the manual pass over a real converted module recorded in the PR.
- The reproducibility invariant is proven behaviorally: after any editor session, running forge's own `assemble()` over the workdir reproduces the session's `adventure.json`, `report.json`, and previews byte-for-byte — the editor added corrections, never hand-edits.
- Every supported op translates to a merged, reasoned, one-entry-per-address `overrides.yaml` that forge's `load_overrides` accepts and whose assembly equals the op-applied document; every unsupported op blocks in place with the detach offer; and a hand-authored overrides file opens, lists, canonicalizes, and extends without loss beyond the documented comment normalization.
- The five shared lint checks are proven in parity by running forge's `check()` beside `lint_adventure` — the citation-based contract phase 2 pinned is retired by the running comparison.
- Notes survive re-keys, removes, and undo without content loss; view state resumes a correction session where it left off; dismissal marks persist exactly as long as the flags they answer.
- Every phase 5 item in the spec's roadmap entry is traceable to code and tests here, or explicitly deferred above with its phase named — including the phase 4 skip's boundary, recorded in the intro.
