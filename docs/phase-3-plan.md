# Phase 3 plan — keyed content

Implementation plan for phase 3 of [the osr-editor spec](spec.md). Phase 3 delivers the map-first stocking flow — the right-click stocking menu, the keyed-entry area cards, stocking badges with next-unstocked navigation, type-ahead pickers — over the full content surface: encounters, features, traps, treasure, and wandering, plus the catalog routes that feed the pickers and publish-to-osr-web with its checkout-path config. Milestone: hand-author a small complete adventure — keyed encounters, a trapped treasure cache, a secret door — publish it, and play it through in osr-web.

Five research facts shape this plan more than any others. First, **the room-vs-treasure trap split is enforced by the owning models, not by `TrapSpec`**: `AreaSpec` rejects any non-room trap ("area {id!r} carries a non-room trap", `osrlib/crawl/dungeon.py:456-461`) and `FeatureSpec` rejects any non-treasure trap ("feature {id!r} carries a non-treasure trap", `dungeon.py:385-390`) — so the UI never asks for a trap's kind; where the trap lives decides it, and the whole-document re-validation backstop turns a mismatched kind into `op_rejected` with osrlib's own message. Second, **because ops embed osrlib content models, those models' internal validators fire at request parse, not at op application**: `KeyedMonster`'s "exactly one of count_dice or count_fixed is required", `AreaTreasureSpec`'s "names letters or sets unguarded, not both or neither", `TrapEffect`'s "a volley needs per-projectile damage dice" and "a condition duration needs a condition", `EncounterTable`'s "rows must cover d20 rolls 1-20 in order" — an invalid combination is a 422 `request_invalid` before any op logic runs, so the phase 3 forms are shaped valid-by-construction wherever shape can carry the rule (radio choices for the XORs, dependent enablement for the implications), and the two value-level rules shape cannot carry — the dice grammar, and `variant_dice`'s span-must-match-pool-size check on monster rows (`osrlib/core/tables.py:348-358`) — get client-side convenience mirrors with the server's parse as the authority. Third, **cross-reference problems stay diagnostics, exactly as the spec's tier rule demands**: `template_id` resolution and pinned-alignment membership are `validate_adventure` checks (`osrlib/crawl/adventure.py:180-190`) the phase 1 parser already maps (`encounter_unknown_monster`, `encounter_alignment_invalid`) — the pickers constrain what the editor authors, and the ops admit what foreign documents legally carry. Fourth, **`validate_adventure` never checks treasure-type letters** — `AreaTreasureSpec.letters` is an unconstrained `tuple[str, ...]`, and the valid letters are whatever `load_treasure_tables()` ships (A–V, 22 types today) — so the letter picker serves the shipped set through the catalogs route and the editor simply cannot author an unknown letter; foreign letters persist untouched (osrlib's own posture), and upstreaming the check is recorded as an end state alongside the parser's other candidates. Fifth, **a committable inline wandering table needs all twenty rows at once** — rolls 1–20 in order, each row a non-empty name, an entry, and exactly one count form — so blind row-by-row authoring cannot produce a valid table, and the runtime consumes an inline table wholesale (`level.wandering.table or load_encounter_tables().for_level(level.number)`, `osrlib/crawl/exploration.py:558`) without ever consulting its level band. The table form therefore starts from the compiled band table for the level (`for_level`'s printed-band clamp, `osrlib/core/tables.py:505-513`) and edits from there — which is why the encounter-tables catalog route ships this phase. A sixth fact sizes two decisions: the shipped catalogs are small (233 monsters, 51 pickable equipment items), so the catalog routes serve whole lists and the pickers filter client-side, and phase 3 adds **zero Python dependencies** — publish is stdlib symlink/copy over the existing config and store machinery.

## Scope

In scope:

- `ops.py` growth — the content ops (`SetEncounter`, `SetTrap`, `SetTreasure`, `AddFeature`, `SetFeature`, `RemoveFeature`), joined into `AnyEditOp`
- `documents.py` growth — op application for the new vocabulary, precise delta pointers
- Catalog routes: `GET /api/catalogs/{monsters|equipment|treasure-types|encounter-tables}`
- Publish: `config.py` growth (`osr_web_checkout`), new module `publish.py`, `POST /api/projects/{id}/publish`, its error vocabulary
- Frontend — module-notation formatters; the catalogs client cache; vendored command/context-menu/popover/checkbox/radio-group components and the type-ahead pickers (monsters, equipment, treasure types); the keyed-entry area cards (encounter, treasure, the trap builder, features); the map stocking layer (hollow/filled key numbers, content glyphs, hover summaries, the unstocked filter, previous/next-area keys, the stocking context menu); level-properties growth (the wandering d20 table editor, level-scope features); the publish dialog
- The committed small-complete-module round-trip fixture (the spec's fixture pair, content half)
- Type generation updates and the test suites for all of the above, green in CI

Out of scope (each with the phase that picks it up): the monster editor and bundled-template authoring — the monster picker searches the effective catalog (shipped SRD plus whatever bundled templates a foreign document carries), but creating and editing templates is phase 4; forge-backed projects entirely, including op→override translation for the new content ops (phase 5); per-area author notes — the sidecar field's consuming surface is this phase's area panel, but notes keyed by entity address need the same re-key/rename/renumber cascade and undo interplay that sidecar view state needs, and shipping them without that design orphans a note the moment its area is re-keyed — silent loss this project refuses; they land with phase 5's sidecar machinery, where persisted view state already forces exactly that work. Conversion (phase 6); SRD stocking, the context menu's roll-stocking entry, treasure/encounter preview, and the prose assistant (phase 7 — the menu ships without the entry rather than with a disabled teaser, the phase 1 rule); walk mode (phase 8); docs site and release engineering (phase 9). The spells and classes catalogs arrive with their consumers (phases 4 and 7); the drag-from-palette toolbox is explicitly not required for 1.0 (spec). Phase 2's stranded-feature honesty note resolves itself here without new work: `ResizeLevel` offenders that are features become fixable because the feature ops can move or remove them.

## Work items

### 1. Content ops

New frozen ops in `ops.py`. Encounter, trap, and treasure ops extend a new `AreaOp` base (`LevelOp` plus `area_id`); the feature ops address a *container* — `area_id: str | None`, where `None` means the level itself — because `LevelSpec.features` is real (`dungeon.py:494`), and an op vocabulary that cannot reach level-scope features leaves foreign documents' feature findings (`feature_needs_cell`, `feature_cell_out_of_bounds`, `feature_id_conflict`) permanently unfixable. Every targeting miss — unknown dungeon, level, area, or feature id — is `op_target_not_found`, per the phase 2 rule.

- **`SetEncounter`** — `encounter: KeyedEncounter | None`; `None` removes. Whole-value grain, the `SetWandering` precedent: the card commits a complete encounter, and internal validity (at least one monster line, exactly-one count per line) is the model's own, enforced at request parse.
- **`SetTrap`** — `trap: TrapSpec | None` on the area. The op does not inspect `kind`: the card builder authors `kind="room"` by construction, and a mismatch is exactly what the re-validation backstop exists to catch — `op_rejected` carrying osrlib's "area {id!r} carries a non-room trap".
- **`SetTreasure`** — `treasure: AreaTreasureSpec | None`. The letters-xor-unguarded rule is the model's, at request parse.
- **`AddFeature`** — container plus `feature: FeatureSpec`. Rejects with `op_invariant`: an empty id; an id already used on the level — `validate_adventure`'s uniqueness scope spans the level's own features and every area's (`adventure.py:160-164`), and a duplicate is never intentional, the `CreateArea` reasoning; the reserved id `"pile"` (colliding with the runtime's drop-pile convention is never intentional either — same posture as the monster-id collision rule the spec pins). Rejects an out-of-bounds `cell` — the editor never authors out-of-bounds geometry (the phase 2 rule); foreign documents' out-of-bounds cells remain diagnostics, and `ResizeLevel` already enumerates them as offenders. Admits `cell=None` at level scope: `feature_needs_cell` is a content finding, legal while editing — a feature can exist before its cell is picked — though the phase 3 UI always creates level features with a cell in hand.
- **`SetFeature`** — container, `feature_id`, and `feature: FeatureSpec`: whole-value replacement; a differing `feature.id` is a rename, under `AddFeature`'s id rejections. The cell-bounds invariant applies to a *changed* cell only: a `SetFeature` whose `cell` equals the targeted feature's current cell passes it through untouched, so editing any other field of a foreign feature with an out-of-bounds cell never locks — the cell stays a navigable diagnostic — while a `cell` that differs from the current value must be in bounds or `None` (re-binding to the whole area). The invariant the ops enforce is thus "no op ever *introduces* out-of-bounds geometry", the exact posture phase 2's `SetEdges` delete rule took toward foreign malformed keys, and the same carry-through the encounter card gives an out-of-intersection alignment pin. Among foreign duplicate feature ids, `SetFeature` and `RemoveFeature` resolve first-match in authored order — osrlib's own resolution posture, phase 2's `RemoveTransition` reasoning. Nothing in the document references feature ids, so a rename cascades nowhere. **This deliberately implements the spec's `SetFeatureField` slot at whole-value grain**: a field-grained op over `FeatureSpec` would need a value union spanning `str | Position | None | tuple[str, ...] | Coins | tuple[ValuableSpec, ...] | TrapSpec` — undiscriminable on the wire and a pairing validator far past what `SetTownField` justifies — while the whole-value form matches `SetEncounter` and `SetWandering`, the card's commit gesture produces the whole model anyway, and the spec's vocabulary is explicitly representative ("and kin. The vocabulary grows additively").
- **`RemoveFeature`** — container plus `feature_id`; first-match among foreign duplicates, like `SetFeature`. Moving a feature between containers (area to level, area to area) is a `RemoveFeature` + `AddFeature` batch — one gesture, one undo step, the transition-edit precedent.

Feature keys are auto-assigned by the card flow as `feature-<n>` — the smallest positive `n` unused in any `feature-<n>` id on the level, the next-free convention area keys established — and stay editable through `SetFeature`.

Delta pointers: area-addressed ops → `/dungeons/<i>/levels/<j>/areas/<k>`; level-scope feature ops → `/dungeons/<i>/levels/<j>/features`. The indexed area pointer is finer than the `/areas` the existing `_replace_area` helper answers for phase 2's area ops — the content ops return the indexed form themselves (a pointer choice, not a helper change; `applyDelta` already walks array indices). Application in `documents.py` rides the existing `_replace_area`/`_replace_level` helpers; the whole-batch re-validation round-trip stays the single model-validity enforcement point, and every op-level check above has it as backstop.

### 2. Catalog routes

Read-only routes over osrlib's shipped data, behind the auth dependency like everything:

```text
GET /api/catalogs/monsters          # CatalogMonster: id, name, page, categories,
                                    #   alignment_options, usual_alignment, hit_dice
GET /api/catalogs/equipment         # CatalogItem: id, name, item_type, cost_gp
GET /api/catalogs/treasure-types    # CatalogTreasureType: letter, kind (hoard/individual/group)
GET /api/catalogs/encounter-tables  # the six compiled dungeon EncounterTables
```

- The summaries are editor-defined route models wrapping what the pickers need; `hit_dice` reuses osrlib's `MonsterHitDice` (the frontend formats it), and `alignment_options` flattens `AlignmentSpec.options` — the field the encounter card's alignment select is constrained by. Equipment flattens the four pickable lists (`weapons`, `armour`, `gear`, `ammunition` — the same four `EquipmentCatalog.get` resolves; `treasure_weights` is not id-addressable equipment and stays out). Treasure types come from `load_treasure_tables().treasure_types`; encounter tables are `load_encounter_tables().tables` verbatim (already generated types, since `EncounterTable` rides in `WanderingSpec`) — the NPC generation tables stay server-side, no consumer.
- Whole catalogs, no server-side search: 233 monsters and 51 items are trivially client-filterable, and a type-ahead over a local list is instant. The routes are project-independent and immutable per process (osrlib's loaders are `functools.cache`d), so the frontend fetches each catalog once per session through a module-level promise cache in the API client — no state-library ceremony.
- The *effective* monster catalog is a client-side merge: the shipped list plus the open document's `monsters` tuple (bundled templates, foreign-authored until phase 4), bundled entries ranked first per the spec's picker contract. No per-project catalog route exists, because the document is already in hand.
- The spec's route sketch (`/catalogs/{monsters|equipment|spells|classes|tables}`) is representative, not exhaustive: spells and classes wait for their consumers, and "tables" lands as the two concrete needs — treasure types and encounter tables — rather than a grab-bag route.

### 3. Publish to osr-web

`config.py` grows `osr_web_checkout: str | None` — additive schema, absent until first use. There is no settings screen: the publish dialog collects the path when unconfigured, the request carries it, and the backend validates and saves it — one route, no dead surface (the phase 1 no-dead-keys rule).

New module `publish.py`:

- **The shape test**: the checkout path must be an existing directory containing an `adventures/` directory — osr-web's discovery root, whose gates phase 1 verified (`adventures/*.json` or `adventures/<dir>/adventure.json`, symlinks followed, rescanned per request). Failure → 422 `osr_web_checkout_invalid` with the remedy naming what an osr-web checkout looks like. The path is validated and saved to config before the destination attempt, so a later collision never costs the user their typed path.
- **The gate**: publish requires the validation tier empty — current diagnostics are recomputed on every commit, so the check reads fresh state. Non-empty → 409 `publish_blocked` with `details.findings` carrying the blocking findings, which the dialog renders with the diagnostics panel's own click-to-navigate rows. Lint never blocks server-side; the frontend confirms when lint findings exist — secret-only access is sometimes the point, the spec's own example.
- **Modes**: `symlink` (default) — `adventures/<name>` → the project directory, so osr-web lists `adventures/<name>/adventure.json` and every always-saved commit republishes live; the `editor.json` sidecar rides along harmlessly (osr-web reads only its own gates). `copy` — the current document's `dump_adventure` bytes written to `adventures/<name>.json`, a point-in-time snapshot.
- `<name>` defaults to the project directory's stem (`my-module.osr` → `my-module`), editable in the dialog; a request-model validator rejects empty names, path separators, and the `.`/`..` forms (surfacing as `request_invalid`, the phase 1 amendment's channel).
- **Collisions**: publish probes both candidate forms (`adventures/<name>` and `adventures/<name>.json`); any existing entry → 409 `publish_destination_exists` unless the request sets `overwrite: true`. Overwrite clears *both* candidate forms before writing the requested one — a mode switch must not leave the other form behind, or osr-web's digest-deduped scan lists a stale snapshot beside the live entry the moment their bytes diverge — but never removes a non-symlink directory: that is somebody's content, refused with the path named. Re-publishing symlink-mode onto a symlink already resolving to this project is idempotent success, no overwrite required.
- `PublishResult` carries the destination path and mode. Route: `POST /api/projects/{id}/publish` with `{mode, name?, overwrite?, checkout_path?}`. Errors: `osr_web_not_configured` (422, remedy: provide the checkout path), `osr_web_checkout_invalid` (422), `publish_blocked` (409), `publish_destination_exists` (409).
- The route is honestly local, like export: a symlink into a user checkout is the single-user posture, and a hosted future replaces this one handler; noted in its docstring. Publish writes nothing inside the project — the destination write is the licensing rule's explicit user-invoked exception, same as export.

### 4. Type generation

The new ops ride `AnyEditOp` into `OpBatch` — free. The catalog models and `PublishResult` are route-referenced. The injection list stays exactly `[ApiError]`; the drift gate is unchanged. `frontend/src/types/index.ts` grows aliases for the content models the cards touch (`KeyedEncounter`, `KeyedMonster`, `FeatureSpec`, `TrapSpec`, `TrapEffect`, `AreaTreasureSpec`, `ValuableSpec`, `Coins`, `EncounterTable`, `EncounterTableRow`, `MonsterHitDice`), the new ops, and the catalog and publish models. The vitest type-level suite extends: the union discriminates over the grown vocabulary, `SetEncounter.encounter` admits `null`, `FeatureSpec.cell` is nullable, `CatalogMonster.alignment_options` is a string-literal array.

### 5. Module notation formatters

`frontend/src/lib/notation.ts` — pure functions, vitest-covered, the one rendering vocabulary shared by the cards' collapsed summaries, the map hover line, and the context menu:

- **Counts** render the dice string verbatim or the fixed integer; a monster line is `<count> × <name>` — `3d4 × orc`, `6 × skeleton`. The multiplication form deliberately sidesteps English pluralization (catalog names are singular; naive pluralization lies — "thiefs") while staying monospace-honest module notation; the spec's "3d4 orcs" example illustrates voice, not a rendering contract worth a pluralization engine.
- **Treasure**: `type C`, `types A, C`, `unguarded`. **Coins**: nonzero denominations in pp/gp/ep/sp/cp order (`120 gp, 30 sp`). **Valuables**: count by kind (`2 gems, 1 jewellery`).
- **Traps**: trigger plus the populated effect parts in model field order (`enter: 2d6, save vs. breath for half`; `open: poison needle — save vs. death or dies` for `kills`; the `manual` prose verbatim when it is the only populated field).
- **Features**: kind label plus payload summary (`cache: 120 gp, 2 gems, 1 item — trapped`, `trick`, `custom`).
- **Encounters**: monster lines joined with `+`; **wandering**: `1-in-6 every 2 turns`, with `custom table` appended when an inline table exists.
- A dice-input convenience check mirrors the documented dice grammar (`osrlib/core/dice.py:5-15`) for immediate field feedback — the mirror is convenience, not authority, the phase 2 edge-key-mirror rule: the server's `parse()` at request validation remains the gate. The same mirror's parse feeds the wandering row editor's `variant_dice` span check (item 9).

### 6. Pickers and vendored components

Vendored shadcn additions, committed source under `components/ui/` per the spec: `command` (cmdk), `context-menu`, `popover`, `checkbox`, `radio-group` — new development-time dependencies `cmdk`, `@radix-ui/react-context-menu`, `@radix-ui/react-popover`, `@radix-ui/react-checkbox`, `@radix-ui/react-radio-group` (current stable majors, per the phase 0 version rule). Native selects continue for plain enum fields (stance, save category, condition, alignment, time unit), the phase 2 posture.

- **The monster picker**: a command palette over the effective catalog — bundled templates first, then this session's recently used, then shipped order — filtering case-insensitively on name and id as you type; the count lands inline in the same gesture (a segmented dice-or-fixed input: all digits reads as fixed, anything else as dice with the convenience check). Recently-used is in-memory session state; persisted stocking state is phase 7's sidecar work.
- **The equipment picker**: the same pattern over the 51 items, grouped by `item_type`, cost shown in monospace.
- **The treasure-type picker**: the 22 letters grouped by section (hoard, individual, group), multi-select for the letters tuple.

### 7. The area panel: keyed-entry cards

`AreaInspector` grows into the keyed-entry reading view its phase 2 header comment promised. The inspector aside widens for area selections (`w-64` → `w-96`): the deep forms expand in place per the spec and need the room — still an aside, never a modal.

- Identity stays on top — id, name, then the description in the serif voice, sized for reading aloud (the spec's prose contract).
- Below it, one compact card per content kind in the spec's order — **encounter, treasure, trap, features**. A collapsed card is a one-line module-notation summary (item 5); an empty kind is a single-click add; cards expand in place to edit; removal is the card's own action. Every commit rides `store.commit` builders patching only their own field against the committed document — the phase 1 amendment — one batch, one undo step per gesture.
- **The encounter card**: monster lines (add via the picker, count editable inline, per-line remove — the last line's remove is disabled with a hint pointing at the card's remove-encounter action, since `monsters` has `min_length=1`); the alignment select offers unpinned plus the intersection of every line's template `alignment_options` (an existing pin outside the intersection surfaces as the already-parsed `encounter_alignment_invalid` — legal and navigable, never destroyed by the form); the `aware` checkbox; the stance select over `ReactionResult` plus none.
- **The treasure card**: a letters-versus-unguarded radio (the model's XOR by construction) with the treasure-type picker on the letters side.
- **The trap card** (room traps): the `TrapSpec` builder — kind pinned `"room"` invisibly; trigger (default `enter`); affects; then the composable effect: damage dice, volley dice (enabled only when damage is set — the validator's implication by construction), save (category, modifier, negates/half), the `kills` toggle, condition plus duration (a dice-or-fixed value with a `TimeUnit`, enabled only when a condition is chosen), `fall_feet`, the slide transition (reusing phase 2's transition target picker for dungeon/level/cell/facing), and the `manual` prose fallback.
- **The features card**: the area's features listed by id and summary; each expands to id (editable, rejections surfaced inline), kind, description (serif), cell binding — whole area, or a picked cell via a one-shot map pick like the transition tool's target gesture, coordinates shown in monospace — `item_ids` via the equipment picker, coins (five non-negative steppers), valuables rows (kind, name, value, weight), and, on `treasure_cache` features only, the treasure-trap builder (the same `TrapSpec` builder with kind pinned `"treasure"`, trigger defaulting `open`). The model would admit a trap on any feature kind, but the spec pins treasure traps to caches, so the form offers it only there; a foreign document's trapped trick passes through untouched.

### 8. The map stocking layer

`render.ts` and the map editor grow the spec's at-a-glance stocking state:

- **Key numbers** render hollow for unstocked areas and filled for stocked. Stocked means a non-empty description or any content kind — a described room is a keyed room in module terms before any mechanics land. The remove-area confirm adopts the same predicate, so a described area never vanishes without a confirm.
- **Glyphs** beside the key number, pencil-weight per the design language: encounter; trap — an area trap *or any trapped cache*, since both are traps the party can spring; treasure — an area treasure *or any `treasure_cache` feature*, since a cache is treasure. The hover line states what the glyphs summarize.
- **Hover** grows the area's one-line contents (item 5's formatters) beside the existing cell/edge ref.
- **The unstocked filter**: a toolbar toggle (shortcut `F`) dims stocked areas — reduced-alpha tint and key number; nothing hides, everything stays clickable.
- **Previous/next area**: `[` and `]` walk areas in key order — numeric ids numerically first, then non-numeric ids lexicographically, so foreign ids participate — selecting each area with the inspector following and `ensureVisible` panning it into view. With the filter on, the walk visits unstocked areas only: stocking a big dungeon is a walk through it.
- **The stocking context menu**: right-click on an area cell (the canvas's currently-suppressed `onContextMenu` becomes the trigger) offers exactly what the area can hold, reflecting current state — description; encounter, treasure, and trap each as add or edit-plus-remove; add feature. Menu actions set the selection plus a one-shot card intent the panel consumes: expand that card, start its add flow. Right-click anywhere else does nothing this phase — cell-scope stocking (level features) lives in level properties, and the roll-stocking entry arrives with phase 7, absent rather than disabled. A pure menu-builder function (area state → entries) keeps the menu vitest-testable.

### 9. Level properties: the wandering table and level features

- **The wandering form** (phase 1's, relocated to level properties in phase 2) grows the inline `EncounterTable` editor. An "override the compiled table" action seeds all twenty rows from the compiled band table for the level — `for_level(level.number)`, fetched through the encounter-tables catalog route — then rows edit in place: name, entry (monster: the monster picker in multi-select for `monster_ids`, plus an optional `variant_dice` input; npc party: basic/expert), and the dice-or-fixed count. `variant_dice` carries a validator shape cannot satisfy — the dice must span exactly the pool's size with no multiplier (`tables.py:348-358`), and the shipped band tables the editor seeds from contain such rows (the hydra rows), so editing a seeded pool can invalidate its dice — the row editor computes the span from the convenience mirror's parse and flags the mismatch inline, with the server's `request_invalid` as the honest backstop, not the primary experience. Rolls 1–20 are fixed row labels, so order and coverage hold by construction and the model's validator is satisfied by shape. Removing the override confirms and commits `table: None` — back to the compiled band. Authored tables pin `id` as `<dungeon-id>-level-<n>-wandering` and keep it stable across edits (the builder carries an existing table's `id` and `label` through); `label` is editable, defaulting `"Level <n> wandering"`; `min_level` is pinned 1 and `max_level` `None` — self-description only, since the runtime consumes an inline table wholesale and never consults its band (`exploration.py:558`). All of it commits through `SetWandering` builders patching only their own fields, the phase 1 carry-through rule finally retired by a real editor.
- **Level-scope features**: a features section in level properties with the same feature cards as item 7, cell required by the UI at creation (`feature_needs_cell` stays the diagnostic for foreign documents). This is the surface that makes level-feature findings navigable-to-fixable and stranded-feature resize offenders resolvable.

### 10. The publish dialog

A header action beside export. The flow: checkout path (prefilled from config; input shown when unconfigured, saved once the backend's shape test passes — item 3's rule), name (defaulting the project stem), mode (symlink default with the live-republish behavior stated; copy as snapshot), then commit. A `publish_blocked` answer renders the blocking findings as click-to-navigate rows; `publish_destination_exists` offers the explicit overwrite; a lint-warnings confirm interposes when lint findings exist, listing them. Success toasts the published path.

### 11. The small complete module fixture

`tests/fixtures/small_module.json`, with a builder function and regeneration script exactly like the torture fixture's pattern: one dungeon, two levels joined by a reciprocal stairs pair (the lint budget below allows exactly one finding, so the levels must connect cleanly) — keyed encounters exercising dice and fixed counts, a pinned alignment, `aware`, and a stance; an area treasure by letters and another unguarded; a room trap (a pit: damage plus save); a `treasure_cache` feature carrying coins, valuables, an item id, and a treasure trap; a `construction_trick` and a `custom` feature, one of them level-scope with a cell; a secret door on the only route to a treasure room; wandering with an inline table on level 2; town, hooks, and services filled. Validation-clean, and lint-clean except the one finding it is built to carry — the secret-only treasure room *is* the `secret_only_access` trigger, and that is the point: the spec's own publish rule ("secret-only access is sometimes the point") needs a module that exercises it, so the fixture declares the finding the way the torture fixture declares its built-to-fail cases, and the publish suite proves a warning-bearing module publishes. Round-trips byte-identically. It serves the round-trip stability suite, the golden osrlib-compatibility discipline, the publish tests, doctored-variant diagnostics fixtures, and the e2e milestone — and it is the module the manual osr-web play-through walks.

### 12. Tests

Consolidated from the work items:

- **ops**: apply and reject per invariant — encounter set/replace/clear; trap set and the re-validation backstop proven (a `kind="treasure"` trap on an area answers `op_rejected`, never commits); treasure set/clear; feature add/set/remove at both scopes, duplicate/empty/reserved-`"pile"` ids, the cell-bounds rule (`AddFeature` always rejects out-of-bounds; `SetFeature` rejects a *changed* out-of-bounds cell and carries an unchanged foreign one through), `cell=None` admission at level scope, rename via `SetFeature`, first-match resolution over a foreign duplicate-id pair, unknown container and feature ids as `op_target_not_found`; delta pointers per op; batch atomicity across the grown vocabulary.
- **catalogs**: route shapes and counts against the installed osrlib; auth introspection and typegen drift (automatic).
- **publish**: the gate (findings → `publish_blocked` with details); a lint-warning-bearing module (the small module itself) publishing cleanly — warnings never block server-side; shape-test failures and the not-configured path; both modes' filesystem results (the symlink resolves to the project directory; copy bytes equal `dump_adventure`); name validation including `.`/`..`; the collision/overwrite matrix including the mode-switch both-forms clear and never-replacing a real directory; idempotent re-symlink; config save-on-shape-pass and the `osr_web_checkout` config growth.
