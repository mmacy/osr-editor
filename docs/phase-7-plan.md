# Phase 7 plan — authoring aids

Implementation plan for phase 7 of [the osr-editor spec](spec.md). Phase 7 ships the three authoring aids: SRD stocking with seeded, reproducible re-rolls; treasure and encounter previews (plus the derive-from-HD stat-block helpers phase 4 deferred here); and the prose assistant behind provider detection. All three apply their results as ordinary op batches — undoable, editable, no special state. Milestone: stock a blank level from the SRD tables, punch up its descriptions with the assistant, and publish.

Eight research facts shape this plan more than any others. First, **osrlib ships the stocking primitives, not just the tables** — `roll_room_contents` (the stocking d6 plus the treasure-presence d6, `osrlib/core/treasure.py:568-591`), `generate_treasure`/`generate_unguarded_treasure`/`generate_magic_item`, and `plan_treasure_ref`, all under a documented determinism contract ("generation draws on the `treasure` stream in printed-entry order … so results are reproducible from the stream alone", `treasure.py:22-26`). The only missing piece is the per-room composition procedure, which osrlib's own spec defers with "the generator that consumes them comes later" (`osrlib-python/docs/spec.md:254`) — a deferral of the arrives-with-its-first-consumer kind, and phase 7 is the first consumer. By owner decision, the siblings may grow where it aligns with their direction, so **the stocking procedure lands in osrlib** as an additive 1.3 surface and the editor consumes it like every other rule — retiring the editor spec's "one rules procedure the editor implements" carve-out, whose own text already named upstreaming as the expected end state. The two spec sentences recording that carve-out are amended alongside this plan (same PR, the phase 6 precedent); the pinned dependency range moves in the implementation PR, when the floor actually moves. Second, **the reproducibility seam is osrlib's named-stream RNG**: `RngStreams(master_seed).get(key)` mints independent PCG64 streams whose identity depends only on `(master_seed, key)` (`osrlib/core/rng.py:16-18`), every draw advances the stream in place, and `RngStream.export_state()`/`restore()` round-trips a `{state, inc}` snapshot (`rng.py:73-91`) — exactly the "seeded RNG recorded in the sidecar; re-roll advances it" contract. PCG state is a 128-bit integer, so the sidecar stores it decimal-string-encoded: the sidecar rides every op result into JavaScript, whose numbers silently lose precision above 2^53, and a lossless-by-construction field beats a field that is safe only because nobody echoes it back. There is no counter shortcut — rejection sampling makes the draw-to-state relation data-dependent (`rng.py:185-211`), so the exported state is the only faithful record. Third, **the "wandering tables" of the spec's stocking bullet are the monster-selection tables**: the SRD procedure says dungeon-room monsters "may be selected by hand or rolled on an encounter table" (`srd/Designing_a_Dungeon.md:74`), and osrlib's compiled level-band d20 tables (`load_encounter_tables().for_level`, clamping into printed bands, `osrlib/core/tables.py:505-525`) are those tables. A row's count "overrides the monster description's number-appearing" (`tables.py:386-388`), rows name monsters by id (pools pick uniformly, `variant_dice` selects the hydra form — the crawl's own conventions, `osrlib/crawl/exploration.py:582-594`), and NPC-party rows have no authorable surface at all: `adventure.json` has no party surface (a spec non-goal), so an NPC-party roll surfaces as a follow-up for the author, never a silent re-roll that would distort the printed odds. Stocking therefore never touches `WanderingSpec` and never emits the one blocked content op. Fourth, **B/X treasure routing is spec-writing, not hoard-rolling**: a monster room's treasure is "the treasure type indicated in its description" (`Designing_a_Dungeon.md:78`), and `plan_treasure_ref` splits a `TreasureRef` into lair/individual/group letters (`treasure.py:530-565`) — individual and group letters ride the spawned monsters at play (the crawl's `_generate_carried_treasure`, `exploration.py:2054-2090`), so only the **lair** letters become area treasure; writing more double-counts. Empty and trap rooms with treasure get `AreaTreasureSpec(unguarded=True)`, which generates from the dungeon level's band at play; the model is letters XOR unguarded by validator (`osrlib/crawl/dungeon.py:309-328`). A rolled monster with no lair letters means treasure was indicated but the description says none — RAW, reported as a follow-up, never backfilled from the unguarded band. Fifth, **every op the aids emit translates in forge-backed mode**: `SetEncounter`, `SetTrap`, `SetTreasure`, and `SetAreaField` name/description all have area-override translations (`overrides.py:291-311`), so stocking and prose run in both project types unconditionally, writing reasoned `overrides.yaml` entries through the existing drafter with no special-casing — and since stocking never emits `SetWandering` (fact three), the one blocked content op (`overrides.py:136`) never arises. Sixth, **forge's provider seam is prompt-agnostic but schema-forced JSON**: `generate` "either returns `data` that validates against `request.schema` or raises" (`osrforge/providers/base.py:1-8`) — there is no free-text path, and none is needed, because prose returns as JSON fields exactly the way forge's own survey stage returns `description` and `hooks` today (`survey.py:85-87`). Text-only requests are what makes fixtures replay-grade (the monsters stage's own stated property, `monsters.py:260-262`), `FixtureProvider`/`RecordingProvider` are generic over any request — "recording for the live tooling" is their module's charter and host-app integration tests their spec-named purpose (`fixtures.py:1`, `osr-forge/docs/spec.md:90`) — and the tag grammar is `^[A-Za-z0-9._-]+$` (`base.py:30`). The editor records its own prose fixtures with `RecordingProvider` and replays them with zero forge changes; no osr-forge modification is needed anywhere in this phase. Seventh, **the phase 6 provider surface is consumed as-is**, closing that plan's stated seam: `resolve_provider` → `build_provider` at request time, `ProviderStatus.configured` as the absent-not-broken gate, and the fixtures kind set through the typed route for e2e (`tests/e2e/conversion.spec.ts:126-131`); token usage arrives per call with schema-retry attempts already folded in (`foundry.py:216-225`), so displaying `response.usage` is the whole job. Eighth, **stocking applies server-side while prose applies client-side, for one principled reason**: results land "as ordinary op batches" (the spec's sentence), and the importer precedent of return-payload-then-client-commits exists because the import dialog has interactive adoption choices between load and apply (`importers.py:11-14`). Stocking has no step between the menu click and the content — always-saved philosophy says the click is the intent — so the stock route builds an ordinary `OpBatch` and applies it through `apply_batch` under the project lock, one atomic roll-and-apply, one undo step, the standard 409 discipline. Prose has an interactive step by definition (insert only on explicit accept), so the client commits the accepted text as an ordinary batch. The derive-from-HD helpers, finally, are fully served by shipped tables — `monster_xp` (`tables.py:714-735`), `thac0_for_hd` (`tables.py:591-610`), and `MonsterSaveBand` carrying real `SavingThrows` values (`tables.py:132-144`) — so phase 4's deferral closes with a pure preview route and zero frontend rules mirroring.

## Scope

In scope:

- The osrlib stocking procedure — `stock_area` and its `StockedArea` result in a new `osrlib.crawl.stocking` module, an additive minor release (1.3.0) developed under osrlib's own loop; the editor consumes it
- Sidecar growth: the stocking seed state (master seed plus per-address stream snapshots), the `set_stocking_seed` patch, and the re-key cascade join
- `aids.py` — stocking orchestration (targets, seeds, op assembly), the three preview kinds (treasure, encounter, stat-block derive), and the prose assistant (facts assembly, prompts, request builder)
- Routes: `POST /api/projects/{id}/aids/stock`, `POST …/aids/preview`, `POST …/aids/prose`; the aid error vocabulary
- Frontend: the roll-stocking menu entry and level sweep with the stocking report, preview affordances in the encounter and treasure cards, derive-from-HD in the monster editor, the prose suggestion flow in the area prose editor and hooks form, provider gating via the existing status surface
- The prose fixture recording script and committed prose fixtures; test suites (pytest, vitest, Playwright) green in CI with zero network
- Type generation growth, README and CHANGELOG updates, the two spec-sentence amendments, and retiring the phase 3 breadcrumbs (`frontend/src/map/stocking.ts:83-84`, `frontend/src/lib/catalogs.ts:93-94`)

Out of scope (each with where it goes): walk mode (phase 8). Docs site and release engineering (phase 9). The spells and classes catalog routes — deferred by phases 3 and 4 to "their first real consumer (phase 7's aids, or later)", and phase 7 turns out not to consume them either (the prose facts are names, counts, and notation, not spell lists), so they wait for a real consumer still, re-recorded here. Upstreaming the structural-lint finding models to osrlib — the phase 1 note's other half; no phase 7 consumer, unscheduled. Prose targets beyond area descriptions and adventure hooks (town prose, monster intros) — the request union grows additively when wanted. Per-request sampling knobs (temperature) — forge's adapter sends none by design; prose rides the same conservative settings conversion uses. USD display for prose calls — the spec's sentence is "token usage is displayed", and forge's pricing constants are model-specific conversion-estimate machinery; tokens are the honest unit here. Whole-dungeon generation (maps from nothing) — stocking fills authored geometry; generation stays a future direction. Persisted preview state — previews are pure, and their optional seeds are request-scoped test surface only.

## Work items

### 1. The osrlib stocking procedure

A new module `osrlib/crawl/stocking.py` (the crawl layer because the result is built from crawl content models; core cannot import them), developed and reviewed under osrlib's own `AGENTS.md` loop, released as 1.3.0:

- **`STOCKING_STREAM = "stocking"`** joins the named-stream constants, and **`StockedArea`** (frozen) carries the procedure's whole answer: `contents: Literal["empty", "monster", "special", "trap"]`, `treasure_present: bool`, `encounter: KeyedEncounter | None`, `npc_party: Literal["basic", "expert"] | None`, `treasure: AreaTreasureSpec | None`.
- **`stock_area(level_number, *, catalog, stream, table=None) -> StockedArea`** runs the SRD procedure exactly: `roll_room_contents(stream)`; on `monster`, a d20 row from `table` (the caller's authored override — the editor passes the level's `WanderingSpec.table` when set, mirroring `wandering_check`'s own resolution) or `load_encounter_tables().for_level(level_number)`, with pool/variant selection per the crawl's conventions and the row's count rolled **now** into `count_fixed` — printed modules pin their counts, and a concrete number is what an author can review, preview, and edit; on an NPC-party row, `npc_party` is set and `encounter` and `treasure` stay `None` (nothing authorable exists, and parties carry class treasure, not letters); with treasure present, a monster room gets `AreaTreasureSpec(letters=plan_treasure_ref(template.treasure).lair)` when the lair letters are non-empty and `None` otherwise, and an empty or trap room gets `AreaTreasureSpec(unguarded=True)`. `catalog` is the effective catalog (the caller composes shipped plus bundled, the way `GameSession.effective_monsters` does) so an authored override table naming bundled monsters resolves.
- The draw order is documented and golden-tested the way `generate_treasure`'s is (contents d6, treasure d6 when the row's chance is non-zero, then d20, pool-or-variant draw, count dice), because the sidecar contract downstream depends on it. Traps and specials produce no models — B/X ships example lists as referee prose, not tables — and that boundary is the docstring's: the procedure ends where the referee's design begins.
- Editor dependency: floor bump to `osrlib>=1.3,<2` in the implementation PR. If implementation outpaces the release, the interim is the recorded family posture — a `[tool.uv.sources]` git pin at the merged commit, swapped for the plain range when 1.3.0 publishes, exactly the osr-forge arrangement.

### 2. Sidecar: the stocking seed

- `EditorSidecar` grows one additive field (no version bump, the envelope's own rule): `stocking: StockingState = StockingState()`, where `StockingState` (frozen) is `master_seed: str | None = None` and `streams: dict[str, StreamState] = {}` — `StreamState` mirroring osrlib's `RngStreamState` as `state: str`, `inc: str`, all three decimal-string-encoded (research fact two). `streams` is keyed by area address, the notes pattern exactly.
- **Derivation and advance:** a fresh area stream is `RngStream.from_seed_material(int(master_seed), f"stock:{address}")` — the `stock:` namespace fences future stream kinds; a stored snapshot restores instead. Per-address streams make re-rolls order-independent: re-rolling room 7 never changes what room 9 would have rolled, and a partial sweep doesn't shift the rooms it skipped. After every roll the advanced state persists to the sidecar in the same critical section. The master seed is minted with `secrets.randbits(64)` at first stocking use and persisted before the first draw.
- **`SetStockingSeed`** joins the sidecar patch union (`action: "set_stocking_seed"`, `master_seed: str`): sets the master seed and clears `streams`, because states derived from the old seed are meaningless under the new one. It is typed API surface with no dialog UI (the fixtures-kind precedent) — the e2e suite's determinism lever, and a reproducible-stocking lever for anyone scripting the API.
- **The re-key cascade:** `streams` keys join the exact remap rules notes follow (`_note_remap_rules` — dungeon rename, level renumber, area re-key), with behavior on area removal identical to notes' current behavior; the suite asserts key-for-key parity with `notes` across the whole rule set so the two maps can never drift.
- **Undo does not rewind the seed** — dice-tower semantics, pinned: the sidecar is annotation state outside the undo history, so undoing a stocking batch restores the document while the streams stay advanced, and re-rolling after an undo honestly yields new content. The report surface says so.

### 3. Stocking orchestration and route

`aids.py` owns the editor-side procedure; the route is `POST /api/projects/{project_id}/aids/stock` with `AidsStockRequest{revision, dungeon_id, level_number, area_id: str | None}` — an area id for the single-room roll, `None` for the level sweep.

- **The target predicate is backend-owned**: `area_is_stocked` (any of description, encounter, trap, treasure, features non-empty) — the authority the frontend's `isAreaStocked` already mirrors, the same one-predicate-everywhere discipline as the edge-key math. The sweep targets exactly the areas the unstocked filter shows, in key order. A single-room roll against a stocked area is 409 `aid_target_stocked` (remedy: undo or clear the room first); the frontend never offers it, the typed route still refuses it. A sweep with zero targets is legal and returns an empty report, no revision bump.
- **One invocation, one atomic act, under the project lock**: verify the revision (stale is the standard 409 before any die is cast — a mechanical refusal must not consume seed state), resolve targets (a missing dungeon/level/area reuses `OpTargetNotFoundError`, no twin code), then per target restore-or-derive the area's stream, call `stock_area` (effective catalog composed shipped-plus-bundled; the level's authored wandering table passed through), and assemble ops: `SetEncounter` when an encounter rolled; `SetTreasure` when a treasure spec rolled; on `trap` contents, `SetTrap` with the editor's placeholder — `TrapSpec(kind="room", trigger="enter", affects="triggerer")` over a `TrapEffect(manual=…)` naming the referee's design job in sentence-case editor copy (composing the placeholder is authoring surface, not rules — the rules procedure ended at "this room holds a trap"). Specials and empty rolls produce no ops — an empty room is honestly empty, stays eligible for a future sweep, and keeps showing in the unstocked filter, which is the affordance that re-surfaces the follow-up work. All targets' ops form **one `OpBatch`** applied through `apply_batch` — one undo step for the sweep, the compound-gesture rule — and the advanced stream states plus any newly minted master seed persist in the same critical section. In forge-backed projects the batch rides `_apply_forge_batch` unchanged: every emitted op translates (research fact five), reasons come from the existing drafter, and no new workdir write exists.
- **Response `AidsStockResponse{rolls, result}`**: `result` is the ordinary `OpBatchResult` (`None` when no ops rolled — the store consumes it exactly like an `/ops` response), and `rolls` is one `StockRoll` per target — `area_id`, `address`, `contents`, `treasure_present`, module-notation summaries (`"5 × acolyte"`, `"type C"`, `"unguarded"`), and typed follow-ups (`"special"`, `"npc_party"`, `"trap_design"`, `"no_lair_treasure"`) so the report surface renders what the dice left to the author.

### 4. Previews

`POST /api/projects/{project_id}/aids/preview`, body discriminated on `kind`, response per kind; pure reads — no revision, no document mutation, no sidecar contact. Optional `seed: str | None` on the sampling kinds (server entropy when absent) is request-scoped test surface, the fixtures-kind posture. All three land in `aids.py` over osrlib's generators and tables; nothing rolls client-side.

- **`treasure`**: `letters: tuple[str, ...]` XOR `unguarded_level: int` (mirroring `AreaTreasureSpec`'s validator), `samples: int = 3` (1..10). Each sample generates on a fresh stream (`generate_treasure` per letter, or `generate_unguarded_treasure`), and the response carries per-sample coins, valuables, and magic items with backend-resolved display names plus a total-gp line — the spec's "N example hoards".
- **`encounter`**: takes a `KeyedEncounter` payload verbatim — the card previews what is *being edited*, uncommitted state included — plus `samples`. Per line: resolution against the effective catalog (shipped, bundled, or honestly `unresolved` — a dangling id is diagnostics-legal and previews as such, never an error), the stat summary (AC pair, HD notation, THAC0/attack bonus, movement, morale, XP, number appearing), sampled counts from the line's dice, and per-sample XP totals.
- **`statblock`**: takes a `MonsterHitDice` payload and answers the derived values — `xp` (`monster_xp`), the `thac0`/`attack_bonus` pair (`thac0_for_hd`), and the save band's label plus its actual `SavingThrows` (`monster_save_band_label` + `CombatTables.save_band`). This is the phase 4 deferral landing in its named home: pure preview through a route, never a frontend rules mirror.

### 5. The prose assistant

- **Route** `POST /api/projects/{project_id}/aids/prose`, target discriminated: `{kind: "area", dungeon_id, level_number, area_id}` or `{kind: "hooks"}`. The facts assemble server-side under the lock — area target: the area's name and key, level and dungeon names, encounter lines resolved to monster names and counts, treasure letters or unguarded, trap and feature summaries in module notation, and the adventure's name and description as tone; hooks target: name, description, town summary, dungeon names, existing hooks. The provider call runs **outside** the lock (a generate call takes seconds to a minute; edits stay live), synchronous request/response — the family posture, and the reason conversion needed workers (stages run minutes) doesn't apply. The client may abandon the request (`AbortController`); there is no server-side cancel for a single bounded call.
- **Requests** follow forge's own house pattern: pure `build_prose_request` functions the route and the recording script share, module-constant system prompts (`PROSE_AREA_SYSTEM`, `PROSE_HOOKS_SYSTEM` — drafting discipline pinned in the prompt: read-aloud register, never invent mechanics, mention only stated facts), tags `prose.area-description` and `prose.hooks`, a single `TextPart` of canonically ordered facts (deterministic serialization, because the fingerprint is the fixture key), and schemas `{description: string}` / `{hooks: array[string]}` — prose wrapped in JSON, the survey precedent.
- **Provider**: `build_provider(resolve_provider(…))` at request time, phase 6's machinery untouched. Unconfigured is 422 `provider_not_configured` at the route and an *absent* affordance in the UI (`ProviderStatus.configured`), the spec's "absent (not broken)". A failed generate — forge's `ProviderError` or `SchemaValidationError` — maps to 502 `provider_request_failed` with forge's message verbatim: an upstream failure on a synchronous route is a gateway error, and conversion's session-state channel doesn't exist here.
- **Response**: the draft, `usage` (`input_tokens`/`output_tokens`, displayed per the spec's sentence), and `model_id`. Acceptance is entirely client-side: the suggestion renders beside the current text, and **accept** commits `SetAreaField(description)` or `SetAdventureField(hooks)` as an ordinary batch (translating in forge-backed projects like any prose correction); discard touches nothing.
- **Fixtures**: `scripts/record_prose_fixtures.py` wraps the resolved live provider in forge's `RecordingProvider` and drives the same build-request path over the committed fixture adventures at pinned states, writing `tests/assets/prose/`; a README records provenance and the re-record rule (a prompt, schema, or facts-assembly edit strands fixtures loudly — `FixtureMissError` names the miss — and re-recording is the remedy, forge's own discipline). CI replays through `FixtureProvider`; no live call ever runs there.

### 6. Routes, errors, and types

- New errors: `AidTargetStockedError` → 409 `aid_target_stocked`; `ProviderRequestFailedError` → 502 `provider_request_failed`. Everything else reuses the vocabulary: malformed aid requests are pydantic 422 `request_invalid`, missing targets are `op_target_not_found`, stale revisions are `stale_revision`, unconfigured providers are `provider_not_configured`. Every new route declares the auth dependency, the phase 0 seam.
- Type generation: the aid request/response models are route-referenced and generate automatically; the injection list stays exactly `[ApiError]`. `frontend/src/types/index.ts` grows the aliases (`AidsStockRequest`, `AidsStockResponse`, `AidsPreviewRequest`/`Response`, `AidsProseRequest`/`Response`, `StockingState`); `AnySidecarPatch` picks up `set_stocking_seed` from the union with no hand edit.

### 7. Frontend

- **Stocking**: `stockingMenuEntries` grows the "Roll SRD stocking" entry, offered exactly when `isAreaStocked` is false (one predicate everywhere; the phase 3 breadcrumb retires). `applyMenuEntry` calls the stock route and feeds `result` through the store's existing `applyResult`. The level sweep is a map-toolbar control beside the unstocked filter, confirming with the target count and the one-undo-step note. Both flows land in the **stocking report** (a dialog for sweeps, a compact popover for single rooms): per-area rows with key number, contents, notation summaries, and follow-up badges with honest copy ("Special — describe by hand", "NPC party — author by hand", "Trap — design it in the trap card", "No lair treasure — the monster's description says none"), plus the undo-does-not-rewind-the-dice note.
- **Previews**: the encounter card gains a preview affordance (sampled counts, the stat table, XP totals) and the treasure card a sample-hoards affordance — both pure display inside the expanded card, nothing committed. The monster editor gains derive-from-HD buttons beside the XP, THAC0/attack-bonus, and saves fields, filling form values from the `statblock` preview; the author commits through the phase 4 form unchanged — suggestions fill fields, the honest form stays the authority.
- **Prose**: the area prose editor and the adventure hooks form gain a "Draft with assistant" affordance, rendered only when the provider status (fetched when the surface mounts) reads configured. The suggestion panel shows the draft beside the current text with the token-usage line and accept/discard; accept commits, discard discards, and an in-flight request shows progress with client-side abandon.
- Store and API client grow the three aid calls; the stock path reuses `applyResult` verbatim.

## Definition of done

- The full gauntlet stays green on both toolchains with zero network in CI; the typegen drift gate passes with the injection list still `[ApiError]`.
- The milestone works end to end: deterministically in Playwright (seeded sweep through publish, prose via recorded fixtures), and by hand with a live provider on a real module, recorded in the PR.
- Every deferral to phase 7 is closed and traceable: the roll-stocking menu entry, the sidecar stocking seeds, the preview-route pattern with the derive-from-HD helpers, and the prose assistant's reuse of the phase 6 provider surface — with the spells/classes catalogs' continued deferral re-recorded, not silently dropped.
- Reproducibility is proven, not asserted: identical master seed and action sequence yield byte-identical documents in the goldens; re-rolls advance per-address streams order-independently; undo never rewinds a stream and the UI says so.
- The forge invariant survives the aids: in forge-backed projects every stocked room and accepted draft is an `overrides.yaml` entry through the existing translation, and no editor code writes any workdir artifact.
- No secret touches disk or a response; the prose assistant is absent when no provider is configured, inserts only on explicit accept, and displays token usage on every draft.
- The spec no longer records a rules procedure implemented in the editor: `stock_area` lives in osrlib behind the same import discipline as every other rule, and every phase 7 item in the roadmap entry is traceable to code and tests here, or explicitly deferred above with its home named.
