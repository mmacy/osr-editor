# Phase 6 plan — conversion

Implementation plan for phase 6 of [the osr-editor spec](spec.md). Phase 6 makes the editor the front door to forge's pipeline: new-from-PDF with provider settings, the estimate-and-cost confirmation gate, the conversion chain on a worker thread with live progress and cooperative cancellation, and landing in the review queue — plus the surfaces phase 5 explicitly deferred here: the pipeline view for incomplete workdirs with graphical resume, reruns whose resumed chain contains a model stage, the standalone regenerate-previews control, and `FixtureProvider`-recorded conversion tests. Milestone: PDF to published adventure without touching the CLI.

Eight research facts shape this plan more than any others. First, **`estimate()` runs the real `preprocess()` into the workdir and leaves it warm**: the licensing invariant forbids persisting module text outside the user's workdir, so a temp directory is not an option (`osr-forge/estimate.py:3-7`), and after estimate the workdir's `run.json` reads preprocess `completed` with every other stage `pending` (`preprocess.py:130-140`) — forge's own docstring names `rerun survey` as the human's next step. The editor's confirm-then-run is therefore `rerun(workdir, Stage.SURVEY, provider, on_progress=…)`, never a second `convert()` (which would re-render every page); in fact **no editor path ever calls `convert()`** — estimate-then-rerun covers new conversions, and `rerun(Stage.PREPROCESS)` reads the workdir's own `source.pdf` for everything else (`convert.py:276-279`). The two spec sentences this supersedes — the conversion section's "`convert()` runs on a worker thread" and the roadmap's "subprocess convert" (which the architecture section already contradicts with its worker-thread-and-cooperative-cancellation contract, `spec.md:67`) — are amended alongside this plan (same PR), the phase 5 precedent: the source of truth must not keep recording a shape the contracts rule out. Second, **an estimated-but-unconfirmed workdir *is* an incomplete workdir** — the exact shape phase 5's open gate refuses with `forge_workdir_incomplete`. The estimate step manufactures the artifact the phase 6 pipeline view must handle anyway, so declining the confirm needs no cleanup path: the warm workdir is honestly resumable, and the two flows (new-from-PDF, resume-an-incomplete-workdir) converge on one surface. Third, **progress and cancellation ride `on_progress`, and only stage boundaries exist**: `_run_chain` emits `running` before each step outside any try block and `completed`/`failed` after (`convert.py:91-109`), each stage writes its own `run.json` status, and an exception raised from the callback on a `running` event propagates before the stage runs — completed stages persist, `run.json` stays consistent, and `rerun` resumes from the first incomplete stage, which is precisely the spec's "a cancel raises out of the `on_progress` callback at the next stage boundary." Fourth, **the provider surface is settings-plus-inference**: `FoundrySettings.from_env` reads `OSRFORGE_FOUNDRY_ENDPOINT`, `OSRFORGE_FOUNDRY_DEPLOYMENT`, and optionally `OSRFORGE_FOUNDRY_API_KEY` (`providers/foundry.py:30-32,59-79`); auth mode is inferred (key present → key auth, absent → Entra ID via `DefaultAzureCredential`, which needs the `osr-forge[entra]` extra and raises `ProviderError` with that remedy at client build, `foundry.py:90-96`); the constructor performs no I/O; and a provider is required exactly when the resumed chain contains a model stage (`convert.py:260-263`), with `rerun` validating settings knobs through `ConversionSettings` and rejecting knobs owned by upstream stages (`convert.py:265-274`). Fifth, **zero-network conversion testing is fingerprint-exact replay over committed page renders**: PNG byte-stability across pdfium/Pillow versions is explicitly not forge's contract (`preprocess.py:1-7`), and fixture fingerprints hash the page images (`providers/base.py:76-98`), so recordings replay only against the exact committed renders they were recorded from — forge's own `test_minimod_pipeline.py` fabricates the workdir from committed pages and runs the real stage functions against `FixtureProvider`, and forge's spec names host-app integration tests as `FixtureProvider`'s purpose (`osr-forge/docs/spec.md:90`). The editor vendors forge's minimod assets (~500 KB, wholly original content from the same owner's MIT repo) pinned to the locked forge revision. Sixth, **`CostEstimate` is a frozen dataclass whose contract is the band, not the point value** (`estimate.py:72-107`; ±40% on input tokens, `estimate.py:9-15`), and hosts surface the USD figure as a confirmation — "converting this 48-page module will cost roughly $X" (`osr-forge/docs/spec.md:216`). Seventh, **`render_previews` regenerates the SVGs from the survey and content caches plus `overrides.yaml` alone**, raising `ValueError` when those caches are missing (`assemble.py:1505-1535`) — actionable exactly on a workdir whose content stage completed but which cannot yet assemble, the pre-assemble state phase 5 named when it deferred the control here. Eighth, **the CLI's commented overrides template exists because "hosts own their correction UX"** (`cli.py:136-141`) — the editor's corrections panel is that UX and its serializer does not preserve comments, so the editor never writes the template.

## Scope

In scope:

- `conversions.py` — the conversion-session registry and state machine: worker thread, polling state, cooperative cancellation, per-workdir exclusivity, project-bound reruns
- `forge.py` growth: estimate, chain-run, and preview-regeneration wrappers; provider construction and status probing (the bridge stays the only module importing forge's operational modules)
- `providers.py` — the editor-side provider configuration models: env detection, session overrides held in memory only, the merged effective config; no secrets ever written to disk
- Routes: `POST /api/conversions`, `GET /api/conversions/{id}`, `POST /api/conversions/{id}/run`, `POST /api/conversions/{id}/cancel`, `POST /api/conversions/{id}/previews`, `GET /api/conversions/{id}/previews/{dungeon_id}/{level_number}`, `GET`/`POST /api/provider`; the conversion error vocabulary; the busy-project guard
- The incomplete-workdir open flow: `forge_workdir_incomplete` routes to the pipeline view (per-stage status, resume, rerun, regenerate previews) instead of a dead-end error
- Pipeline-panel growth for open forge projects: model-stage reruns through bound conversion sessions, with adoption of the re-assembled document and the busy guard
- Frontend: the new-from-PDF dialog with the estimate confirmation gate, the conversion screen, the provider settings dialog, the home-screen entry, polling
- Vendored minimod assets, the warm-workdir fabrication helper, and the test suites (pytest, vitest, Playwright), green in CI with zero network
- The two spec-sentence amendments, README and CHANGELOG updates, and retiring the phase 5 comment breadcrumbs (`errors.py:72`, `projects.py:287`, the pipeline panel's "Model-stage reruns arrive with conversion" copy)

Out of scope (each with the phase that picks it up): SRD stocking, treasure/encounter preview, and the prose assistant (phase 7 — the prose assistant consumes this phase's provider configuration and status surface as-is; that reuse is the seam, not new work here). Walk mode (phase 8). Docs site and release engineering (phase 9). Providers beyond Foundry and forge's own `FixtureProvider` — the provider seam is forge's (`ModelProvider`), a new vendor lands in forge first, and the editor's configuration surface follows it then. Persisting estimates or conversion history — sessions are in-memory, the workdir's `run.json` is the durable record, and a server restart loses nothing but a progress view (the workdir resumes). Remote-store conversion (a hosted future syncs workdirs through the materialization seam; the local store's identity materialization ships).

## Work items

### 1. Provider configuration

`providers.py` holds the editor-side models; `forge.py` does the forge-touching work (the seam rule):

- **`ProviderSessionSettings`** — endpoint, deployment, api_key, kind (`"foundry" | "fixtures"`), fixtures_dir — every field optional, held on `app.state` only. The effective config merges field-wise: a session value wins over its `OSRFORGE_FOUNDRY_*` environment counterpart; kind defaults to foundry. Secrets never touch disk: the config file schema does not grow, and the test suite asserts it (a provider-configured session leaves `config.json` byte-identical).
- **`ProviderStatus`** — the `GET /api/provider` answer: kind; endpoint and deployment values with per-field source (`"env" | "session" | None`); `api_key_present: bool` with its source — **never the key bytes, and no response model carries a key field** (type-level test); `entra_available: bool` (azure-identity importable — the inferred auth path when no key is present); `configured: bool` (a provider can be built: endpoint and deployment present, and either a key or Entra availability); fixtures_dir when the kind is fixtures. `POST /api/provider` sets or clears session fields (explicit `null` clears) and answers the new status.
- **`forge.py` grows `build_provider(config) -> ModelProvider`** — `FoundryProvider(FoundrySettings(...))` from the merged config, or `FixtureProvider(fixtures_dir)`; a build-time `ProviderError` — today exactly the Entra-extra case, since the constructor performs no I/O — is re-raised as `ProviderNotConfiguredError` carrying forge's message *verbatim*, so it rides the one `provider_not_configured` code rather than inventing a second: "no usable auth" is precisely a not-configured condition, and forge's message already names its remedy. Missing required fields raise the editor's `ProviderNotConfiguredError` before forge is involved, naming each missing field and both places it can come from (the environment variable, the provider settings dialog).
- **The fixtures kind is typed API surface, not dialog UI.** Forge built `FixtureProvider` for exactly this — "unit tests, CI, and host-app integration tests run the full pipeline with zero network and zero cost" — and the editor's Playwright suite is its consumer, setting it through the same typed route the dialog uses. The settings dialog renders the Foundry fields only; no end-user affordance exists for a kind no end user holds recordings for.
- **The provider settings dialog** — reachable from the new-from-PDF dialog and the pipeline view's provider strip: endpoint and deployment inputs with env-detected placeholders and source badges, a write-only key input (set/clear; presence shown, value never echoed), and the Entra hint when no key is present (names the `osr-forge[entra]` extra). Copy states the posture: session values live in memory until the editor closes; the `OSRFORGE_FOUNDRY_*` environment variables are the durable configuration.
- **The env-var names are editor-owned constants.** Forge exposes them only as private module constants (`_ENDPOINT_ENV` and kin, `foundry.py:30-32`); the documented public mapping is the three names themselves, so `providers.py` hardcodes `OSRFORGE_FOUNDRY_ENDPOINT`/`_DEPLOYMENT`/`_API_KEY` as its own constants and never imports `osrforge.providers.foundry` — the seam rule stays intact, and forge's `from_env` (which raises on a missing var rather than reporting per-field sources) is not the detection tool anyway.

### 2. The conversion session machinery

New module `conversions.py`: a registry of `ConversionSession`s on `app.state`, keyed by server-minted id and indexed by resolved workdir path.

- **A session** carries: id, kind (`"pdf" | "workdir"`), the pdf path (pdf kind), the resolved workdir path, the requested `ConversionSettings` (pdf kind; the run.json echo governs afterward), state, the `CostEstimate` (pdf kind, once estimated), per-stage rows (stage, status, usage — seeded from `run.json`, updated live from stage events, re-read from `run.json` at terminal states so the final rows are the disk's truth), the error message (failed state, forge's message verbatim), the bound project id (sessions created over an open project's workdir), a `threading.Event` cancel flag, and a lock guarding every read and mutation.
- **States**: pdf kind `estimating → estimated → running → completed | failed | cancelled`; workdir kind `ready → running → …`. **Idle is the runnable test** — every state but the two that hold a worker (`estimating`, `running`) admits a run: `estimated`/`ready` are the first, `failed`/`cancelled` are the resume from the first incomplete stage, and `completed` is the pipeline panel's *next* rerun, which item 7 needs by construction (one workdir keeps one session across every stage a human asks it to redo). Terminal sessions stay in the registry for the process lifetime — single-user, bounded by use, and reload-proof.
- **Creation** (`POST /api/conversions`): pdf kind takes `pdf_path`, `workdir_path` (both absolute), optional settings knobs, and `allow_existing: bool = false`; it spawns the worker immediately (state `estimating` — preprocess renders every page, far too slow for a request thread) and the estimate lands in the session. **The destination guard**, the family's overwrite pattern: a nonexistent or empty directory proceeds; an existing forge workdir (`run.json` parses as `RunMeta`) without `allow_existing` is 409 `conversion_destination_exists` with `details.completed` (whether its monsters stage reads `completed`), and with the flag proceeds — re-preprocessing in place is forge's own documented rebuild behavior, and the flag exists so the dialog confirms first with the honest copy (item 6): the *estimate itself* re-renders `pages/` and resets `run.json` to preprocess-only, and superseding a *completed* conversion means the model stages must be paid for again before it reviews. A file the operating system dropped into a directory a human looked at (`.DS_Store` and kin) counts toward none of these tests: it is the OS's, not the user's, it regenerates on its own, and letting one turn "browse the folder before converting into it" into a refusal would be the guard protecting nothing at the user's expense. A directory holding nothing but an *interrupted preprocess's* residue — `source.pdf` and possibly a partial `pages/`, with no `run.json` — rides the same handshake with `details.completed: false`, because the editor caused it: forge copies `source.pdf` in before it opens the PDF, so a rejected source leaves exactly that, and refusing the obvious next act (pick the right file, keep the prefilled destination) as somebody else's content would leave hand-deleting a directory the editor made as the only remedy. A non-empty directory that is anything else is the existing `ProjectExistsError` (409 `project_dir_not_empty` — somebody's content, never touched); a path that is not a directory at all is 422 `conversion_destination_invalid`. Workdir kind takes `workdir_path`, seeds the stage rows from `run.json`, and starts nothing (state `ready`); creating it for a path that already holds a session **returns that session** — the idempotency that keeps one workdir on one session however many surfaces ask. Create-or-return runs *under the registry lock*, the same discipline `get_or_open` uses for projects, and the pdf kind's exclusivity test registers under that lock too: a check-then-act would let two submitted dialogs both miss and spawn two workers rendering into one directory.
- **Recovery is a lookup, not a guess**: `GET /api/conversions?workdir={path}` answers the path's current session (any state) or 404. Every surface that hits a busy conflict routes through it — a 409 `conversion_in_progress` from project open or from pdf-kind creation sends the frontend to the lookup and then the conversion screen, which also covers a page reload during `estimating` (re-entering the new-from-PDF dialog conflicts, looks up, and lands on the live progress); the pipeline panel probes the lookup when a forge project's screen mounts and re-attaches to an active bound session. Reload-proof, restart-honest: after a server restart the lookup 404s, sessions are gone, and the workdir's `run.json` is the durable truth the open flow already handles.
- **Exclusivity**: a workdir holds at most one *active* (`estimating`/`running`) session. pdf-kind creation over a path with an active session, or over an open project, is 409 (`conversion_in_progress` / `workdir_open_as_project` — the latter's remedy names the pipeline panel, whose workdir-kind session covers every rerun including preprocess, since `source.pdf` lives in the workdir). Opening a project whose path holds an active *unbound* session is 409 `conversion_in_progress` — the open's re-assembly would race the chain.
- **Bound sessions**: a session whose path is an open forge project binds to it (the pipeline panel's model-stage rerun). **The binding is re-resolved at every run, not fixed at creation** — one workdir keeps one session for the process lifetime while the project at its path comes and goes, and the mainline flow produces exactly that mismatch: the *pdf-kind* session that just converted a module is still that workdir's session when the finished conversion is opened and a stage is re-run over it. A session that ran with a stale binding would complete correctly on disk and adopt nothing, leaving the open project rendering a document the workdir no longer holds. Binding an active session is left alone; the busy guard is what keeps that safe. One window is accepted knowingly: a project opening *between* the bind and `begin_run` runs once unbound. It needs two simultaneous in-flight requests — once `begin_run` flips the state the open is refused as active-unbound, and once the open registers the next run re-binds — so it is the same request-sized, single-user class as the `_require_idle`/`begin_run` window, left on the `get_or_open` precedent ("single-user, milliseconds, simplicity wins"). Closing it would need adoption-time re-resolution or a lock spanning two registries, and neither buys anything one human at one keyboard can reach. While it is active, every project route that mutates the document or touches workdir artifacts — ops, undo, redo, forge overrides, check, the synchronous assemble rerun, detach, publish, export — answers 409 `conversion_in_progress`; sidecar patches stay legal (annotation state, no workdir contact). Running a bound session with `stage=assemble` is 409 `conversion_state_invalid` naming the synchronous rerun route — one act, one path, never a slower duplicate. On completion the service adopts the chain's assembly through the existing `_adopt_assembly` path: whole-document delta, revision bump, diagnostics recomputed, `checked` cleared — **adopted before the terminal state lands**, because the busy guard reads that state: the instant it is no longer active a poll can refetch the project and a request can commit against it, and both must see the adopted document. On chain *failure* the project keeps its pre-rerun in-memory state, revision unchanged, while the disk caches are whatever the chain wrote before failing — honest and recoverable, because every later assembly (a commit, an undo, a reopen) runs against the new caches and surfaces forge's error through the phase 5 snapshot protocol, which restores on failure; the canonical case is a `custom_monsters: off` rerun invalidating the cache state an existing `monster_templates:` entry requires (phase 5 research fact six), remedied by removing the entry in the corrections panel or rerunning monsters with the pass on. **A rerun is not an undo step and the undo history survives it**: the forge undo stack snapshots `overrides.yaml` bytes and the reason ledger — the document is derived state — so undo after a monsters rerun restores earlier corrections and re-assembles them against the new caches, which is exactly the semantics a cache regeneration should have; where a knob change made a snapshot's entries unsatisfiable, the undo fails loudly with forge's message and restores, never silently.
- **The worker** is one daemon `threading.Thread` per run. The runnable bodies (`run_estimate(session)`, `run_chain(session, stage)`) are plain functions the thread wrapper calls, so pytest drives them synchronously — no sleeping tests, no flaky threads; the threaded path is covered once by the Playwright suite's real server.
- **Running** (`POST …/run`): optional `stage` (validated against `RUNNABLE_STAGES`; default the first incomplete stage from `run.json` — for a freshly estimated pdf session that is survey, per research fact one) and optional settings knobs. **Everything rejectable rejects before the thread spawns**: the stage validates against forge's own `RUNNABLE_STAGES` (422 `forge_rerun_invalid` — geometry has no independent run, and forge's identical guard inside the chain stays the backstop); the knobs validate synchronously at the route — `ConversionSettings` for shape (the `request_invalid` channel, as in phase 5) and `KNOB_STAGES` for the owner guard (422 `forge_rerun_invalid` with forge's own remedy phrasing) — and the provider is built at request time exactly when the resumed chain contains a model stage, with `ProviderNotConfiguredError` a 422 the same way. What remains inside the worker can only fail as session state. Legal from any idle state; a session a worker already holds is 409 `conversion_state_invalid`, and so is a pdf session whose source was rejected before preprocessing ever wrote a `run.json` — there is no workdir to resume, and the honest act is a new conversion, not a retry of this one.
- **Cancellation** (`POST …/cancel`): sets the flag; the `on_progress` closure checks it on `running` events only and raises the editor's `ConversionCancelledError`, so a stage in flight always finishes — the spec's "takes effect at the next stage boundary," and `run.json` never shows a stage the chain abandoned mid-write. The runner catches it, re-reads `run.json`, and lands the session in `cancelled`. A cancel during `estimating` is checked when preprocess returns: the session lands in `cancelled` with the warm workdir kept. During `estimated`/`ready` it is immediate. The UI copy states the boundary semantics.
- **Failure**: any exception out of the worker lands the session in `failed`. Forge's own — `PdfError` from the estimate (encrypted, over a cap, not a module: the wrong-file guard the settings caps exist for), `ProviderError`, `SchemaValidationError`, `ExtractionError`, `OverrideError` from the chain — carries forge's message verbatim. **The catch is `Exception`, deliberately wider than that hierarchy**: the destination is a path the user typed, so an `OSError` (an unwritable parent, a full disk) is entirely reachable, and a worker that died with the session still active would leave the registry claiming the workdir busy forever, refusing every open, retry, and cancel until the server restarts. A session that cannot fail cleanly is worse than one that fails loudly. **Failure is session state, never an HTTP error**: the request that spawned the worker answered long ago, so there is no `pdf_invalid` code and no error-mapping table entry — one channel, the one the dialog and the conversion screen already render.
- **Recents**: the pdf kind records the workdir (type forge, name the PDF stem) when the estimate completes — the moment the directory becomes a real, resumable workdir; the workdir kind records at creation. Both ride the existing `record_recent` path.
- The editor writes no overrides template into converted workdirs (research fact eight), and no editor code writes any workdir artifact — every write inside the session is forge's own, which keeps phase 5's reproducibility invariant intact through conversion.

### 3. The forge bridge growth

`forge.py` adds, with the phase 5 error-surfacing discipline (forge's messages verbatim, never paraphrased):

- `estimate_pdf(pdf_path, workdir, settings) -> CostEstimate` — wraps `estimate()`; `PdfError` propagates untouched, because its only caller is the session runner and a pdf failure is session state (item 2), never an HTTP mapping.
- `run_chain(workdir, stage, provider, settings_updates, on_progress) -> ConversionResult` — wraps `rerun()`, model stages now legal, **with no error mapping**: like `estimate_pdf`, its only caller is the session runner, whose catch set — `Exception`, so that nothing at all can wedge a session in `running` (item 2) — lands every failure in session `failed` with the message verbatim. (The phase 5 assemble-only wrapper keeps its HTTP mappings — it serves a synchronous route; the worker's channel is session state.)
- `render_workdir_previews(path) -> tuple[Path, ...]` — wraps `render_previews`; the route gates on survey and content `completed` (409 `conversion_state_invalid` otherwise), with the `ValueError` → `ForgeWorkdirInvalidError` mapping as the unreachable-through-the-UI backstop.
- `build_provider` / `provider_status` per item 1. New imports, pinned: `estimate` from the facade; `render_previews` from `osrforge.assemble` (the phase 5 plan's noted deferral, consumed now); `RUNNABLE_STAGES` and `KNOB_STAGES` from `osrforge.convert`; `FoundryProvider`/`FoundrySettings` from `osrforge.providers.foundry`; `FixtureProvider` from `osrforge.providers.fixtures`; `ModelProvider` from `osrforge.providers.base`; `CostEstimate` from `osrforge.estimate`; `PdfError`/`ProviderError` from `osrforge.errors`. `convert` stays unimported — nothing calls it (research fact one).

### 4. Routes, state, and errors

- **`ConversionState`** — the poll answer and every conversion route's response: id, kind, state, workdir path, pdf path, the estimate (forge's `CostEstimate` dataclass riding the OpenAPI surface directly — pydantic serializes stdlib dataclasses; no hand-written mirror), the ordered stage rows (`stage` plus forge's whole `StageStatus` — state, error, timestamps, and `TokenUsage`, embedded rather than flattened, so the conversion screen and the pipeline panel render one shape from one component), the error message, the bound project id. `GET /api/conversions/{id}` answers it (404 `unknown_conversion`, the restart remedy matching `unknown_project`); `GET /api/conversions?workdir={path}` is the recovery lookup (item 2). Every new route — conversions, previews, provider — declares the auth dependency, the phase 0 seam.
- **New errors**: `ConversionNotFoundError` → 404 `unknown_conversion`; `ConversionDestinationExistsError` → 409 `conversion_destination_exists` with `details.completed` (the `allow_existing` handshake, item 2); `ConversionDestinationInvalidError` → 422 `conversion_destination_invalid` (the path is not a directory); `ConversionStateInvalidError` → 409 `conversion_state_invalid` (run or cancel in a state that does not admit it, previews before the caches exist, `stage=assemble` on a bound session); `ConversionInProgressError` → 409 `conversion_in_progress` (the busy guard, both directions); `WorkdirOpenAsProjectError` → 409 `workdir_open_as_project` (remedy: the pipeline panel); `ProviderNotConfiguredError` → 422 `provider_not_configured` (details listing the missing fields). The occupied-directory case reuses `ProjectExistsError` (409 `project_dir_not_empty`) — same refusal, same remedy, no twin code; a stage with no independent run reuses `ForgeRerunInvalidError` (422 `forge_rerun_invalid`), which is already "forge's rerun would reject this". `ConversionCancelledError` is internal control flow and never crosses the API.
- **`ForgeWorkdirIncompleteError`'s remedy** changes from the CLI sentence to the graphical one ("Open the workdir to resume the conversion in the pipeline view") — the code and 422 stay, because the API contract is unchanged; what changes is that the frontend now catches this code and routes to the conversion screen instead of a toast. The `errors.py` and `projects.py` phase 6 breadcrumbs retire with it.
- The synchronous `POST /api/projects/{id}/forge/rerun` stays assemble-only — the correction loop's fast path needs no session — and its docstring drops the "in phase 5" qualifier: model stages have a home now, and it is the conversions API.

### 5. The pipeline view for incomplete workdirs

The frontend flow the spec pins ("a workdir whose conversion never completed opens into the pipeline view … rather than review"):

- **Project-shape detection relaxes to `run.json` alone.** Phase 5's predicate required `run.json` *plus* an artifact under `stages/`, on the reasoning that an empty `stages/` was unreachable for a workdir forge had written into. The estimate step makes that false: an estimated-but-unconfirmed workdir holds `run.json`, `source.pdf`, and `pages/` with no stage caches at all, and it must classify forge or the open answers `not_a_project` instead of `forge_workdir_incomplete`. Shape, never loadability (the module's own rule): a `run.json` that does not parse still classifies forge, so the open surfaces `forge_workdir_invalid` with its repair remedy.
- Opening a path that answers 422 `forge_workdir_incomplete` (home screen recents, open-by-path, `osr-editor PATH`) routes to the **conversion screen** for that workdir: `POST /api/conversions {kind: "workdir"}` (idempotent per the registry rule), then render. A 409 `conversion_in_progress` from any open routes the same way through the recovery lookup — the screen attaches to the live session instead of dead-ending on a toast.
- **The conversion screen** — one component serving both entry flows: the per-stage rows styled like the pipeline panel (status, timestamps' honest absence, token usage), the provider strip (status summary plus the settings-dialog affordance), the stage picker defaulting to the first incomplete stage, the optional knobs row (the pipeline panel's KEY=VALUE pattern), the run control (confirm copy names the model stages the chain will run and that cost is not re-estimated on resume — the estimate belongs to the pdf flow's gate), the cancel control with the boundary-semantics copy, and the failure card (forge's message verbatim, re-run affordance).
- **Regenerate previews** appears once survey and content read `completed`: `POST …/previews` then the preview dialog over `GET …/previews/{dungeon_id}/{level_number}` — the phase 5 forge-preview dialog pointed at the conversion-scope route, so a mid-pipeline cancel can eyeball geometry before spending on the remaining model stages. (For *open* projects the phase 5 posture stands: every commit re-assembles and previews follow; the standalone control lives only here, where no assembly can run.)
- **Completion navigates**: `POST /api/projects/open` on the workdir path, landing in the review queue with the report's flags — the phase 5 chrome, unchanged.
- While a session is `running`, the screen polls `GET /api/conversions/{id}` at one second — the spec's polling posture, and the interval the progress view's stage granularity deserves (stages run minutes, not milliseconds).

### 6. New-from-PDF

- The home screen grows the **"Convert a PDF"** entry beside new/open (the spec's home-screen list has carried it since phase 1's design).
- **The dialog**: PDF path (absolute, the export/import posture), destination workdir (prefilled `<pdf-dir>/<pdf-stem>.forge` on PDF-path entry — the CLI's default, editable), the provider strip, optional knobs. Estimate (`POST /api/conversions`) shows the estimating state, then the **estimate card**: page count, per-stage token rows, and the cost as forge's contract phrases it — "roughly $X", with the rough-estimate framing (the band, not the point value, is the contract). Two actions: **Convert** (`POST …/run`, then the conversion screen) and **Not now** — which closes the dialog with the honest note that the rendered workdir is kept at its path and resumes from the home screen. No estimate, no run: the gate is structural, not advisory.
- An existing-workdir destination rides the `allow_existing` handshake: the first request answers 409 `conversion_destination_exists`, the dialog confirms with the accurate copy — the estimate itself re-renders the pages and resets `run.json` to preprocess-only, and when `details.completed` is true, that this workdir is a finished conversion whose model stages would have to be paid for again before it reviews — then retries with the flag. The warning names what the *estimate* does, not just the run, because declining after the estimate cannot undo it.
- Failure surfaces forge's `PdfError` message verbatim in place (encrypted, over the page cap, not a PDF) — the wrong-file mistake caught before any model spend, exactly what the settings caps are for.

### 7. Model-stage reruns for open projects

The pipeline panel's promise ("Model-stage reruns arrive with conversion") lands:

- The panel's rerun control grows the stage picker (all of `RUNNABLE_STAGES`); assemble keeps the synchronous route (the server rejects a bound `stage=assemble` run, item 2 — one act, one path), any other stage creates the bound session (`POST /api/conversions {kind: "workdir"}` on the project's path) and runs it with the chosen stage and knobs — the monsters-knob cases the forge contract names (`custom_monsters`, `monster_fuzzy_threshold`) become graphical here.
- The bound session lives on the **project store**, probed and polled by a watcher mounted for the whole forge-backed project screen rather than by the panel itself — the panel unmounts on every section change, and the progress view and the commit pause must both outlive that. A page reload mid-rerun re-attaches through the recovery lookup instead of orphaning the run.
- While the session is active the panel shows the inline progress rows and cancel, the chrome disables commits, and the busy guard answers 409 for anything that slips through — the frontend renders it as the panel's busy state, never a toast loop.
- On completion the panel reflects the adopted document: refreshed report, run, diagnostics, cleared `checked`, bumped revision — the poll's terminal state carries the signal, and the frontend refetches project state.
- The confirm copy states the consequences a cache rerun has that an override edit does not: report flags may change wholesale (dismissal marks keyed to vanished flags go dormant, the phase 5 rule doing its job); the undo history replays corrections against the new caches rather than restoring the old document; and when the knobs include `custom_monsters: off`, existing printed-notation patches will fail the chain's assembly until they are removed or the pass is rerun (item 2's failure aftermath, stated before the spend instead of after it).
