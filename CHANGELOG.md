## [Unreleased]

### Added

- Phase 6 — conversion: the editor is the front door to forge's pipeline. **Convert a PDF** on the home screen prices the run first — forge's `estimate()`, which really preprocesses into the destination workdir (module text is never persisted outside your own project) — and presents the page count, per-stage token predictions, and a rough-cost card as a structural gate: no path reaches a running conversion without passing through it, and declining keeps the rendered workdir, which the home screen lists and the pipeline view resumes. Confirming resumes the chain on a worker thread with per-stage progress polled live, cooperative cancellation that lands on stage boundaries (the stage in flight always finishes, so `run.json` never records an abandoned stage and every cancelled conversion resumes), and forge's own message verbatim on failure; success opens straight into the review queue. A workdir whose conversion never completed now opens into the **pipeline view** — per-stage table, stage picker defaulting to the first incomplete stage, `knob=value` settings, and a regenerate-previews control where the survey and content caches exist but assembly cannot yet run — instead of refusing with a CLI remedy. Open projects gain model-stage reruns from the **Pipeline** panel through a session bound to the project: assembly keeps its synchronous fast path, every other stage runs with progress and cancellation and adopts the re-assembled document, commits pause while it runs, and the undo history survives, replaying corrections against the new caches. **Provider settings** reads the `OSRFORGE_FOUNDRY_*` environment the forge CLI reads, reports per-field values and sources plus Entra availability, and accepts session overrides held in memory only — no credential is ever written to editor config and no API response carries one. New `POST/GET /api/conversions`, `…/run`, `…/cancel`, `…/previews`, and `GET/POST /api/provider`; the conversion suites and the Playwright milestone run with zero network against forge's `FixtureProvider` over vendored minimod recordings.

- Phase 4 — the monster editor: the always-present Monsters section (both project types) stocks `Adventure.monsters` with bespoke templates — create from scratch over a model-valid stock 1-HD seed, or clone-and-modify any catalog monster through the new `GET /api/catalogs/monsters/{id}` full-template route, with the picker's "Create monster…" shortcut starting the same flow from the encounter card; the always-saved detail editor covers the whole `MonsterTemplate` (the AC pair gated by the attack-roll toggle, the hit-dice builder, ordered attack routines, movement modes, saves, morale and alternates, alignment, XP notes, dice/fixed/see-below number appearing, the treasure reference, abilities with typed params, defenses, categories) committing whole-value `set_monster_template` batches; an id introducing a collision with the shipped catalog or another bundled template rejects at commit with an inline rename prompt (foreign documents already carrying one stay editable, the finding navigable to the new `monster:<id>` address); a rename cascades every keyed-encounter line, wandering row, and author note in one undo step; removing a referenced template confirms with the reference count and leaves honest diagnostics. In forge-backed projects the three ops block in place with the detach offer — flow entries open the dialog client-side, and the import dialog's new-level mode returns as a visible choice that does the same (the phase 5 absence retired); the forge nav's resolution entry relabels to "Monster resolution". The small-module fixture bundles a bespoke template keyed in an encounter, putting `Adventure.monsters` under the round-trip, golden, and publish discipline permanently.

- Phase 5 — forge-backed projects: opening an osr-forge workdir assembles it fresh and lands in the review chrome — the review queue over `report.json`'s flags with per-flag dismissal marks, source pages beside the inspector, and forge's SVG previews from the map toolbar; every committed edit translates to a merged, reasoned `overrides.yaml` entry (page-anchored machine drafts until a human composes, the explicit-wall seal for synthesized openings, the growth rule for drawing past the derived extent) and re-assembles through forge's own pure loop, with undo/redo as snapshot pairs of the overrides file and the reason ledger; the corrections panel surfaces the reviewable record with inline reason editing and per-entry removal; the monster-resolution panel offers catalog remaps and printed-notation stat-block patches (exclusive per name); the pipeline panel renders `run.json` per-stage status and token usage with the on-demand playability check (findings wipe on the next change, by forge's design) and the assemble-stage rerun knob; ops with no override kind block in place with the detach offer, and detach writes a provenance-recorded native project with author notes carried over; publish and export use forge's bytes verbatim, so a symlink publish republishes live on every correction. The `editor.json` sidecar grows persisted view state, per-entity author notes with the re-key cascade (both project types), review marks, and the machine-draft ledger; the diagnostics envelope gains the forge tier; and the cross-implementation lint-parity suite runs forge's `check()` beside the editor's lint on the shared fixtures. (Phase 4, the monster editor, is deferred by owner decision.)

- Phase 3 — keyed content: the map-first stocking flow — the right-click stocking menu, hollow/filled key numbers with encounter/trap/treasure glyphs, module-notation hover summaries, the unstocked filter (`F`) and the `[`/`]` key-order walk — over the full content surface: keyed-encounter, treasure, room-trap, and feature cards in the area panel with type-ahead pickers fed by the new `GET /api/catalogs/*` routes (monsters, equipment, treasure types, encounter tables), the composable trap builder (kind pinned by where the trap lives), the inline d20 wandering-table editor seeded from the compiled level-band table, level-scope features, the content op vocabulary (`set_encounter`, `set_trap`, `set_treasure`, `add_feature`, `set_feature`, `remove_feature`), and publish to osr-web (symlink or snapshot into a checkout's `adventures/`, gated on validation only — lint warns, with a confirm).

- Phase 2 — the map editor: the graph-paper canvas with the full geometry tool set (rooms, corridors, walls and doors in every state, areas, entrance, transitions with auto-reciprocal stairs), multi-level and multi-dungeon management with rename/renumber cascades and offender-listing resize, the live structural lint tier (forge's `edge_invalid`, `area_unreachable`, `orphan_cell`, `secret_only_access`, and `transition_unpaired` mirrored exactly, plus the editor-only `area_overlap`) rendered on the map with click-to-navigate down to cells and edges and a one-click fix for invalid foreign edge keys, and the `GeometryImporter` entry-point seam with import-from-project as its first implementation — imported levels land as one undoable op batch.

- Phase 1 — projects and prose: the home screen (recents, new adventure, open by path, `osr-editor PATH`), native project create/open with the `editor.json` sidecar, the document service (atomic op batches with revisions and 409 stale-revision safety, bounded undo/redo, always-saved persistence), live content-validation diagnostics with click-to-navigate addresses, the adventure/town/hooks/wandering forms in the pencil-palette design language, the open-time fidelity warning for documents a newer osrlib wrote, and export of the stamped document to any path.
- Phase 0 scaffolding: the wired FastAPI + Vite skeleton (`osr-editor` serves the built frontend and `/api/status`), canonical adventure serialization with a committed round-trip golden fixture, the locked ops/revision/diagnostics envelope, the auth dependency seam, the `ProjectStore` protocol with a local filesystem store, and pydantic→TypeScript type generation with a CI drift gate.
